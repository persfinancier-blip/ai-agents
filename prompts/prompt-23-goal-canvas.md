# Промпт №23: Шаг Ф4 — канвас постановки цели (декомпозиция + увязка)
### `feat/frontend-goal-canvas`, `feat: goal canvas — radial decomposition and KPI links`

> **Кому:** Claude Code. «Круговая постановка цели»: канвас, где вокруг цели собирается вся её постановка — KPI, подцели, связи. Первый экран, показывающий граф увязки KPI→KPI (API есть с Шагов 3a/3b/3c, но его ещё никто не видел).
> **Модель/режим:** Sonnet, обычный режим. Backend НЕ менять: всё нужное уже в API (`/goals`, `/goals/{id}/subtree`, `/kpi-links`, `/kpi-links/cycles`, `/kpi-factors` — проверь роуты перед началом).
> **Канон:** `Visual_Reference.md` Часть II (as-built v2) + Часть III (выход «зум в узел → постановка/декомпозиция»); `Management_Model.md` §9 (структура ≠ смысловая увязка, судья цикла); ADR-0004 (типы связей; визуализация — его открытый вопрос, этот проход закрывает его ДЛЯ КАНВАСА, не для общей карты).
> **Предусловие:** №22 смёржен. «Массовые модификации» в `git status` без правок — протухший кэш (DEVLOG №16).

## Scope

**Делаем:**
1. **Вход и выход.** В `RealGoalCard` — действие «канвас постановки»; открывается `GoalCanvas.tsx` (новый компонент, путь держит `App.tsx` рядом с существующим `goalPath`). «Назад» — к карточке; `Escape` — тоже. Демо-карточки это не касается.
2. **Радиальная композиция** (на полотне v2: фон, сетка, плавающие пилюли):
   - в центре — цель (крупный узел: имя, владелец, бейдж туман/определена);
   - верхний сектор — **KPI цели**: узлы-таблетки (имя, target+unit; композитный — значение с пометкой «расчёт» и пунктирными рёбрами-факторами к своим факторам);
   - нижний сектор — **прямые подцели** (мини-узлы как на карте; клик — переход к карточке подцели);
   - слева — владелец; справа — чек постановки: три строки-проверки из уже существующих данных (владелец есть? KPI измеримы? связи вверх есть?) в стиле ok/warn (светофор статусов).
   - Раскладка — простая тригонометрия по секторам, без force-directed; при переполнении сектора — дуга в два ряда. Это временная раскладка, как на карте.
3. **Связи KPI→KPI на канвасе** (ядро прохода):
   - рёбра от KPI этой цели к KPI других целей (обе стороны: исходящие и входящие, `?kpi_id=`), тип кодируется цветом/подписью: `contributes` / `constrains` / `depends_on` (+ мини-легенда); внешний KPI рисуется маленьким «спутником» с именем своей цели, клик — переход к той цели;
   - **создать связь**: у KPI действие «связать…» → поповер: выбор KPI другой цели (список целей → их KPI) + тип → `POST /kpi-links` (400/409 — русские ошибки);
   - **разорвать связь**: клик по ребру → «разорвать» с подтверждением → `DELETE`;
   - **циклы**: если KPI цели состоит в цикле (`/kpi-links/cycles`) — подсветить ребро/узлы, показать плашку «цикл увязки · судья: <имя цели-судьи или —>» и переключатель «подтверждён» (`PATCH /kpi-links/cycles/{id}`, семантика ответственности, не прав).
4. **Факторы композитного KPI** — read-only: пунктирные рёбра «фактор → композит» с весами; управление факторами (`POST/DELETE /kpi-factors`) — НЕ здесь, отдельный срез (зафиксировать TODO-строкой в BACKLOG).
5. После мутаций связей — перезагрузка данных канваса (без оптимизма, как в Ф3).

**НЕ делаем (сознательно):**
- Backend не трогать. Управление факторами композитов — нет (только просмотр).
- На ОБЩЕЙ карте связи по-прежнему не рисуем — открытый вопрос ADR-0004 закрывается только внутри канваса; в ADR-0004 добавить пометку об этом решении (2–3 строки, не переписывать).
- Никакого drag-n-drop, сохранения координат, force-directed.
- Процесс достижения (переходы seq/split/join) — НЕ здесь: модели на бэке нет, это следующая пара «ADR + бэкенд-срез → канвас процесса».
- Редактирование полей цели на канвасе — нет (для этого карточка, Ф3).
