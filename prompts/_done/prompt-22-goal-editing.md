# Промпт №22: Шаг Ф3 — редактирование целей из UI
### `feat/frontend-goal-editing`, `feat: goal editing from UI (create/patch/delete)`

> **Кому:** Claude Code. Первый функциональный срез поверх шкуры v2: карта перестаёт быть read-only.
> **Модель/режим:** Sonnet, обычный режим. Backend полностью готов (`POST/PATCH/DELETE /goals`, каскад `?cascade=`, diff-sync KPI по стабильным id, валидация циклов) — **backend не менять вообще**.
> **Канон:** `Visual_Reference.md` Часть II (as-built v2) — только существующие токены/классы; UX-решения владельца (2026-07-14): **создание — двойным кликом по полотну**, **правка — инлайн, без форм-оверлеев**.
> **Предусловие:** №21 смёржен. «Массовые модификации» в `git status` без правок — протухший кэш (DEVLOG №16).

## Scope

**Делаем:**
1. **Создание — двойной клик по пустому полотну.** В точке клика появляется черновик-узел со сразу активным инлайн-вводом имени: `Enter` → `POST /goals` (только имя; owner пуст, KPI нет → цель честно рождается в тумане), `Escape`/пустое имя → отмена без запроса. Работает и при пустой БД (двойной клик по демо-карте создаёт ПЕРВУЮ реальную цель — карта переключается на реальный режим). **Координаты клика не сохраняются** — их поля нет в модели; после сохранения узел встаёт по автораскладке (черновик до сохранения рисуется в точке клика). Зафиксировать это ограничение комментарием, хранение позиций не изобретать.
2. **Инлайн-правка в карточке цели (`RealGoalCard`).** Клик по значению превращает его в ввод, `Enter`/blur → `PATCH`, `Escape` — отмена:
   - имя, описание, владелец (`owner` пустой → «отв. —», подсказка «без владельца цель в тумане»);
   - **KPI-строки**: добавить (имя + target + unit), изменить, удалить; отправлять ВЕСЬ список `kpis` с сохранением `id` существующих (diff-sync бэкенда живёт по id — не терять их!); композитные KPI (`computed_value != null`) показывать как сейчас, `target` им не редактировать;
   - **родитель**: строка «родитель: …» — выбор из списка целей (простой поповер-список по `listGoals`, без поиска) или «сделать корневой» (`parent_id: null`); ответ 409 (цикл) → русская ошибка «Нельзя: цикл в дереве», состояние не менять;
   - **бэклог**: переключатель «на карте / в бэклоге» (`is_backlog`) — из бэклога узел исчезает с карты (существующее поведение `buildGoalForest`).
3. **Удаление.** В карточке действие «удалить цель» с подтверждением; ответ `has_children` (409) → второй вопрос «Удалить со всеми подцелями (N)?» → `DELETE ?cascade=true`; после удаления — возврат на карту.
4. **Синхронизация карты.** После любой мутации — перезагрузка `listGoals` (без оптимистичных обновлений и кэшей — просто и предсказуемо); определённость пересчитывает бэкенд, туман на карте обновится сам (это и проверяем: добавил owner + KPI с target → узел вышел из тумана).
5. Ошибки сети/валидации — короткие русские сообщения в духе существующих состояний, без новых компонентов-тостов (строка статуса в карточке/у черновика).

**НЕ делаем (сознательно):**
- Backend не трогать (включая сид и миграции). Связи KPI→KPI и факторы (`/kpi-links`, `/kpi-factors`) из UI не управляются — отдельный срез.
- Никакого drag-n-drop узлов и сохранения координат; раскладка остаётся автоматической и временной.
- `lifecycle_stage`, `risk_level`, `role_label` не редактируем (их UI-семантика не канонизирована).
- Демо-карточку `GoalCard.tsx`, `data.ts`, `goals.ts` не трогать; демо-режим остаётся read-only (кроме двойного клика, создающего первую реальную цель).
- Советник целей не участвует (LLM-постановка — отдельная веха); никаких новых форм-оверлеев.

- UI-текст — русский. Стили — только существующие токены/классы v2; расхождения — в «Открытые вопросы» Части II.
