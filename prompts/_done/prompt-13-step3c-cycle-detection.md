# Промпт №13: Шаг 3c — детект циклов графа связей KPI→KPI
### `feat/goal-step3c-cycle-detection`, `feat: detect and flag cycles in the kpi link graph`

> **Кому:** Claude Code, по образцу Шага 3a (`app/models/kpi_link.py`, `app/services/kpi_link_service.py`, `app/api/v1/kpi_links.py`, `app/schemas/kpi_link.py`).
> **Канон:** `docs/full-vision/02_Product/Management_Model.md` §9.11 (направленность, циклы, судья), `docs/adr/0004-goal-link-types-cycles-composite-kpi.md` (Р3 + «Последствия»).
> **Решение, зафиксированное владельцем перед этим промптом** (ADR-0004 явно оставил это «на момент написания промпта», см. «Последствия», последний пункт): обнаруженные циклы **персистятся** в отдельной таблице `kpi_link_cycle`, пересчитываемой diff-sync'ом при каждом изменении графа связей — тем же паттерном, что уже используется для KPI цели (Шаг 3-0, `goal_service.sync_kpis`) и предписан ADR-0002 §7.7 («пересчёт на каждое изменение»). Это даёт ручному флагу «подтверждён как ставка» стабильную привязку к конкретному циклу.

## Scope

**Делаем:**
- Таблица `kpi_link_cycle`: обнаруженные элементарные циклы графа `kpi_link` как персистентные строки с ручным флагом `confirmed`.
- Пересчёт цикла при каждом `create_link`/`delete_link`/`delete_links_for_kpi` (те же три точки мутации графа, что уже существуют в `kpi_link_service`).
- Вычисление «судьи» — ближайшего общего структурного предка (по `goal.parent_id`) целей-владельцев KPI, входящих в цикл.
- Эндпоинты: список текущих циклов + подтверждение/снятие флага.

**НЕ делаем (сознательно, не домысливать):**
- Циклы **не запрещают** создание связи — `create_link` остаётся как есть (уже реализовано в Шаге 3a, комментарий `# Cycles are not checked here` в коде — этот комментарий теперь устареет, см. ниже).
- Визуализация циклов и типов связей на карте — открытый вопрос ADR-0004 «Открытые вопросы» п.1, фронт-срез, не эта задача.
- Модель прав на подтверждение цикла — как и `is_backlog` (Шаг 1), эндпоинт подтверждения БЕЗ проверки роли/прав («судья» — семантика ответственности для человека, не enforced access control; §7 вопрос 3 ADR-0002 остаётся отложен).
- Соотношение структурной свёртки ресурсов и смысловой увязки (Management_Model §9.8) — отдельный открытый вопрос, эта задача его не закрывает.

## Модель данных

`app/models/kpi_link_cycle.py`, по образцу `kpi_link.py` (не Entity-подтип — тот же аргумент: узлы графа это KPI, циклу не нужен полноценный Entity):

```python
class KpiLinkCycle(Base):
    __tablename__ = "kpi_link_cycle"

    id: Mapped[str]                          # uuid, PK
    canonical_key: Mapped[str]                # см. "Каноническая идентичность" ниже — уникальный индекс
    member_kpi_ids: Mapped[list[str]]         # JSON, id узлов-KPI по кругу в порядке обхода (JSONType, см. app/db/types.py)
    member_link_ids: Mapped[list[str]]        # JSON, id рёбер kpi_link, образующих цикл, в том же порядке
    confirmed: Mapped[bool]                   # default False
    judge_goal_id: Mapped[str | None]         # id цели (goal) — ближайший общий структурный предок; None если предка нет
```

Миграция: `create_table`, без ALTER существующих таблиц (как в Шаге 3a — batch-режим SQLite не нужен).

## Каноническая идентичность цикла (для diff-sync)

Цикл — это последовательность KPI-узлов, замкнутая в кольцо. Один и тот же физический цикл может быть найден обходом с разных стартовых узлов и в двух направлениях перечисления — идентичность должна быть устойчива к этому:

- Нормализовать список `member_kpi_ids`, начиная с лексикографически наименьшего id узла (ротация списка).
- `canonical_key` = нормализованный список id узлов, соединённый разделителем (например `"->"`).join) — используется для сопоставления «этот цикл уже существовал / это новый / этот исчез» при пересчёте.

Если в графе есть параллельные рёбра между одной и той же парой KPI разных типов (`contributes` и `constrains` одновременно, раздел 9.10 допускает множественность) — считать это отдельными различными циклами по факту использованных рёбер, дублирования по одним и тем же узлам не разрешать искусственно; это редкий крайний случай, специально не оптимизировать.

## Алгоритм обнаружения

Полный граф `kpi_link` небольшой (тот же аргумент дешевизны, что и ADR-0002 §7.7) — пересчитывать **все** элементарные циклы графа целиком при каждой мутации, не инкрементально:

- DFS от каждого ещё не обработанного узла-KPI, ведя список текущего пути (стек id).
- При переходе по ребру в узел, уже присутствующий в текущем пути — найден цикл: срез пути от этого узла до конца плюс замыкающее ребро.
- Собрать все найденные циклы со всех стартовых узлов, нормализовать (см. выше) и дедуплицировать по `canonical_key`.

Написать как чистую функцию `find_cycles(links: list[KpiLink]) -> list[Cycle]` в `kpi_link_service` (аналог `would_create_cycle` в `goal_service.py` по духу — читаемая, тестируемая отдельно от сессии БД), затем сервисная функция `sync_cycles(session)`, вызываемая в конце `create_link`, `delete_link`, `delete_links_for_kpi`:

- Посчитать текущий набор циклов через `find_cycles`.
- Сопоставить с существующими строками `kpi_link_cycle` по `canonical_key`: совпадающие — оставить как есть (флаг `confirmed` сохраняется), отсутствующие в новом наборе — удалить, новые — создать с `confirmed=False`.
- Пересчитать `judge_goal_id` для новых/оставшихся строк (см. ниже) — можно пересчитывать всегда, это дёшево.

Обновить докстринг `create_link` (`"""Cycles are not checked here..."""`) — он больше не точен: связь по-прежнему не блокируется циклом, но теперь he запускает пересчёт `kpi_link_cycle`.

## Судья цикла

Для набора `member_kpi_ids` цикла:
1. Для каждого KPI найти его `goal_id` (владеющую цель, `kpi.goal_id`).
2. Для каждой такой цели построить цепочку предков по `goal.parent_id` до корня (переиспользовать логику обхода из `goal_service.would_create_cycle`/`get_subtree` — не копировать код бездумно, но паттерн обхода тот же).
3. Ближайший общий предок — первый узел, общий для всех цепочек (пересечение с сохранением порядка от ближайшего к дальнему).
4. Если общего предка нет (узлы цикла в несвязанных ветках дерева, либо один из KPI сам является корнем без родителя) — `judge_goal_id = None`. Это легитимное состояние, не ошибка — фронт/владелец решает вручную, кому эскалировать; в API просто отдаётся `null`.

## Схемы и роут

`app/schemas/kpi_link.py` (или новый `kpi_link_cycle.py`, по аналогии с существующим разделением):

```python
class KpiLinkCycleRead(BaseModel):
    id: str
    member_kpi_ids: list[str]
    member_link_ids: list[str]
    confirmed: bool
    judge_goal_id: str | None
```

`app/api/v1/kpi_links.py` — добавить:

- `GET /kpi-links/cycles` → `list[KpiLinkCycleRead]`, список всех текущих циклов.
- `PATCH /kpi-links/cycles/{cycle_id}` с телом `{"confirmed": bool}` → `KpiLinkCycleRead`, 404 если цикла с таким id нет. Без проверки прав (см. Scope).

Роуты `/kpi-links/cycles` должны идти в коде ДО `/kpi-links/{link_id}`, если используется общий префикс — проверить, что FastAPI не спутает `cycles` с параметром `{link_id}` (стандартная ловушка порядка роутов).

## Тесты

Интеграционные (`test_kpi_links.py` или новый `test_kpi_link_cycles.py`):
- Цикл из 2 узлов (A→B, B→A, оба `contributes`) — появляется в `GET /kpi-links/cycles` после создания второй связи.
- Цикл из 3 узлов — аналогично.
- Удаление одной из связей цикла (`DELETE /kpi-links/{id}`) → цикл пропадает из списка (строка `kpi_link_cycle` удалена).
- Подтверждение цикла (`PATCH .../confirm`) сохраняет `confirmed=True`, пока состав цикла не меняется (создание несвязанной третьей связи не сбрасывает флаг).
- Разрыв и повторное образование того же цикла (удалить ребро → пересоздать) → `confirmed` сбрасывается в `False` (новая строка, новая идентичность) — тест, явно фиксирующий это поведение, оно неочевидно.
- Судья: KPI цикла принадлежат целям с общим предком (использовать фикстуру дерева из тестов Шага 2b) → `judge_goal_id` = id этого предка.
- Судья: KPI цикла принадлежат несвязанным веткам → `judge_goal_id = None`.
- Каскад: удаление KPI, входящего в подтверждённый цикл (через `delete_kpi`), убирает и связь, и сам цикл из `kpi_link_cycle` (переиспользует существующий каскад `delete_links_for_kpi`, который теперь должен также запускать `sync_cycles`).

Юнит-тесты (`test_kpi_link_service.py` или новый `test_kpi_link_cycles_service.py`):
- `find_cycles` — пустой граф, граф без циклов, один цикл, два независимых цикла, нормализация даёт одинаковый `canonical_key` независимо от стартовой точки обхода.
