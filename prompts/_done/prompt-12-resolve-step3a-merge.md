### `chore`-проход · разрешение 2 конфликтов (BACKLOG.md, DEVLOG.md) · тесты зелёные после merge-коммита

> **Кому:** Claude Code. **Не создавать новую ветку и не делать `git merge` заново** — merge уже запущен на `main` (`git status` покажет `You have unmerged paths`, `.git/MERGE_HEAD` присутствует). Задача — довести ЭТОТ merge до коммита, не начинать заново.
> **Контекст:** ветка `feat/goal-step3a-kpi-links` (граф связей KPI→KPI, Шаг 3a) сливается в `main`. Весь код (модели/схемы/сервисы/роут/миграция/тесты) уже staged БЕЗ конфликтов — трогать код не нужно. Конфликты только в двух доковых файлах, потому что параллельно им обеим правила ветка `docs/plan-fact-and-clients` (уже слита в `main` раньше).

## Проверка перед началом

```bash
git status
```
Ожидается: `Unmerged paths: both modified: BACKLOG.md, both modified: docs/DEVLOG.md`; все остальные файлы (`backend/...`) уже в `Changes to be committed` без конфликта. Если картина другая — остановиться и сверить с владельцем, не гадать.

## Конфликт 1 — `BACKLOG.md`

Конфликт вокруг одной и той же задачи «Шаг 3», описанной двумя способами:

- **Сторона HEAD** (старая формулировка) — один пункт «Шаг 3» целиком (граф связей + детект циклов вместе), ещё не отмеченный как сделанный.
- **Сторона `feat/goal-step3a-kpi-links`** (входящая) — та же работа уже раздроблена по факту реализации: `[x] Шаг 3a` (граф связей, сделано) отдельно от `[ ] Шаг 3c` (детект циклов, ещё не сделано).

**Решение: оставить целиком входящую сторону (`feat/goal-step3a-kpi-links`), сторону HEAD убрать.** Это не «объединение», а замена устаревшей неразбитой формулировки на актуальную — входящая версия точнее отражает состояние репозитория (3a реализован и слит по коду, который сейчас перед вами в staged-изменениях). Убрать маркеры `<<<<<<< HEAD` / `=======` / `>>>>>>> feat/goal-step3a-kpi-links` вместе со строкой HEAD между ними.

Остальной файл (всё, что не в конфликте) не трогать.

## Конфликт 2 — `docs/DEVLOG.md`

Конфликт из двух РАЗНЫХ записей (не дублей) — журнал append-only, обе описывают реальную, но разную работу:

- **Сторона HEAD** — запись `## 2026-07-08 · docs/plan-fact-and-clients` (план/факт + клиентская архитектура).
- **Сторона `feat/goal-step3a-kpi-links`** — запись `## 2026-07-13 · feat/goal-step3a-kpi-links · не закоммичено` (граф связей KPI→KPI).

**Решение: сохранить ОБЕ записи полностью, упорядочив по дате — новые сверху** (правило журнала, шапка файла). Значит `## 2026-07-13 · ...` должна оказаться ВЫШЕ `## 2026-07-08 · docs/plan-fact-and-clients`, хотя в конфликте сейчас они в обратном порядке. Убрать маркеры конфликта, физически переставить блоки.

Дополнительно: запись `2026-07-13` сейчас озаглавлена «не закоммичено» — после того, как этот merge будет закоммичен, это уже неверно. Поправить заголовок записи на `## 2026-07-13 · feat/goal-step3a-kpi-links` (убрать «· не закоммичено»), содержание записи (что сделано / дальше) не менять — это фактический отчёт о работе, он остаётся как есть.
