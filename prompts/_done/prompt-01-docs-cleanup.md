## Задача F — Зафиксировать модель управления (НОВЫЙ канон)

Согласовать с PRD (при конфликте — приоритет PRD, конфликты вынести отдельным списком в конце документа). Mermaid-диаграммы приветствуются. Содержание:

### Ф1. Два типа карт
- **Дерево-декомпозиция** — рекурсивная майнд-карта: цель дробится на подцели/этапы/задачи. Узлы связаны отношением «содержит» (PRD §14.4). На каждом узле — **ролевой ярлык владельца**: собственник / менеджмент / исполнитель (это ярлыки, НЕ жёсткие уровни). Вложенность свободная.
- **Воркфлоу-навык** — отдельный тип карты на листьях дерева: BPMN-подобный граф шагов с ветвлением (split), объединением (join) и условиями переходов. Стиль — как n8n.
- Это ДВА разных полотна, не зум одного. Проваливание: узел дерева → карта уровнем ниже; лист → карта навыка.

### Ф2. Роли и поток управления
- Собственник ставит стратегическую цель с числом (напр. прибыль 1 000 000).
- Менеджмент раскладывает на этапы-драйверы (выручка, закуп, накладные…).
- Исполнитель превращает лист в операционную задачу, назначает **юнит** (человек или ИИ-сотрудник) и прикрепляет **навык-воркфлоу**.
- Роли — ярлыки на узлах, декомпозиция свободна по глубине.

### Ф3. Увязка (reconciliation) — мягко-жёсткая
- Система (ИИ-советник) **предлагает** декомпозицию цели на KPI и требуемые ресурсы; человек утверждает/правит.
- Требования сверяются с ресурсным слоем: **Персонал** (люди), **Финансы** (активы/пассивы), **прочие ресурсы** по усмотрению советника (не всё меряется деньгами).
- Разрыв (напр. выручка 3 000 000 при активах 500 000) **подсвечивается, но НЕ блокирует**. Узел живёт в одном из состояний:
  - **Туман войны** — не определён (нет KPI/владельца/данных).
  - **Бэклог** — определён, но осознанно отложен.
- Оба состояния видимы и сигналят «здесь надо доработать». (В прототипе уже есть визуальные заготовки «туман войны» и «разрыв увязки».)

### Ф4. Увязка через опрос офисных агентов
- **Агент-советник** при постановке цели декомпозирует её и **сам маршрутизирует** запросы к профильным офисам (никаких зашитых правил).
- Офисы: **фронт** (рынок/продажи/клиенты), **мидл** (процессы/производство/операции), **бэк** (финансы/HR/ресурсы). Один драйвер может опрашивать несколько офисов.
- Офис-агент возвращает **аналитическую справку**, не «да/нет»: достижимо (да/частично/нет), реальный потолок, в чём и на сколько дефицит, что нужно, чтобы закрыть.
- Советники работают на ВСЕХ уровнях декомпозиции (стратегия/тактика/операции), с разной глубиной.
- В прототипе каркас уже есть: `demo.ts` содержит `ADVISORS` со скоупами `advisor/front/middle/back` — задача будущих промптов оживить их.

### Ф5. Навык ≠ Компетенция
- **Навык** — устойчивый переиспользуемый воркфлоу (набор шагов → результат), сущность на карте навыков (Ф1). Пример: «парсинг ТГ-каналов → оформление в Google Sheets».
- **Компетенция / оценка** — насколько набор навыков юнита покрывает требования задачи. Если у ИИ-агента есть «парсинг», а задача требует ещё «рассылку» — система показывает разрыв: «агента надо доучить» → создаётся новый навык-воркфлоу.
- Терминологически развести жёстко: в текущем прототипе «навык» с пипсами 0–4 — это на самом деле **компетенция**, переименовать при реализации.

### Ф6. Всё есть Сущность
- Цель, этап, задача, навык, юнит, ресурс, офис-агент, справка — узлы единого графа знаний (PRD §14–§15). Карты — представления подграфов Живого графа.
