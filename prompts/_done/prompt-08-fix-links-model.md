## Решения к фиксации (приняты владельцем)

**Р1. Множественность и типы связей.** Один KPI может ссылаться на несколько вышестоящих KPI. У связи есть **тип**. Стартовый набор — **три**:
- **способствует** (contributes) — вклад в ценность вышестоящей цели (ремонт → статус);
- **ограничивает** (constrains) — конфликт за ресурс/бюджет (ремонт → прибыль как бюджет);
- **зависит-от** (depends-on) — предусловие/вход: цель не может двигаться без другой (продажи ← отчёт себестоимости).
Новые типы добавляются ТОЛЬКО под конкретный сценарий, не влезающий в три (зафиксировать это правило явно, чтобы не плодить типологию).

**Р2. Удаление KPI-источника.** KPI — источник истины для связи. При удалении KPI **связь исчезает как сущность** у всех потребителей снизу (метрика стала неактуальной). Связь каскадно удаляется вместе с ЛЮБЫМ своим концом (источник или потребитель). Никаких «осиротевших» ссылок, никаких запретов удаления.
- **Следствие (записать):** это делает переход replace-all → **diff-sync** в `patch_goal` обязательным до появления связей — иначе патч цели молча удаляет связи вместе с пересоздаваемыми KPI. (Уже в BACKLOG; здесь — зафиксировать причинную связь.)

**Р3. Направленность и циклы.**
- Связи **разнонаправленные**; направление несёт тип («способствует» обычно вверх; «зависит-от» — в любую сторону, в т.ч. вбок/вниз). Правила «только вверх» НЕТ.
- Циклы в графе связей **не запрещаются**, а **детектируются и подсвечиваются** как риск «работа по кругу / работа ради работы».
- **Разрешить/разорвать цикл может только ближайший общий структурный руководитель над всеми узлами цикла** — то есть ближайший общий предок узлов цикла по дереву `parent_id` (Шаг 2b). Он либо подтверждает цикл как осознанную ставку (НИОКР, разведка — «прямой связи с прибылью нет, но а вдруг выстрелит»), снимая тревогу, либо разрывает. Это ручное решение поверх авто-детекта — та же философия, что ручной «бэклог» (Шаг 1).
- **Ключевой смысл:** граф связей ставит вопрос (нашёл цикл), дерево структуры назначает ответственного (кто вправе решить). Две оси, намеренно разведённые, здесь работают вместе.

**Р4. Качественные KPI → составной KPI.** Качественная цель («гудвилл», «статус», «новый офис»), у которой нет прямого числа, раскладывается на **факторы-KPI с весами**; её оценка = взвешенная сумма факторов.
- Пример: гудвилл = 0.3·(динамика отзывов) + 0.3·(LTV) + 0.4·(динамика брака); новый офис = 0.3·аренда + 0.3·удалённость + 0.4·депозит.
- **Составной KPI = KPI, чьё значение собрано из дочерних KPI через веса** (KPI уже сущность, Шаг 2a — новой сущности не нужно).
- Правило тумана расширяется: узел «определён», если есть таргет **ИЛИ** есть формула из факторов (каждый фактор сам измерим).
- Характер: **«решение в моменте»** — применяется тогда, когда конкретную качественную цель надо разложить. Это обогащение, **отдельный Шаг 3b**, после связей.

## Задача A — Расширить §9 Management_Model

Дописать в §9 (новыми подразделами, напр. §9.9–§9.12, не ломая §1–§8 и существующие §9.1–9.8):
- **Типы связей** (Р1) — три типа с определениями и примерами; правило «новые типы только под сценарий».
- **Жизненный цикл связи** (Р2) — связь живёт, пока живы оба её KPI; удаление любого конца удаляет связь; ссылка на BACKLOG-пункт про diff-sync с объяснением причины.
- **Направленность и циклы** (Р3) — разнонаправленность, детект+подсветка, «судья = ближайший общий структурный предок узлов цикла», флаг «осознанная ставка».
- **Составной KPI** (Р4) — факторы с весами, расширение правила тумана, пометка «Шаг 3b».
- Обновить Mermaid-диаграмму §9: показать разные типы рёбер связи (способствует/ограничивает/зависит) хотя бы схематично; добавить мини-пример цикла с «судьёй».
- Обновить §9.8: пункт про «две оси агрегации» дополнить, что теперь у связей есть типы (ограничивает vs способствует), и это влияет на будущую визуализацию.
