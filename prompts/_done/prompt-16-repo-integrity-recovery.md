## Scope

**Делаем:** бэкап → диагностика → лечение индекса/commit-graph → полный прогон проверок → DEVLOG. Плюс два хвоста прошлого прохода (см. ниже).

**НЕ делаем:**
- Никаких правок контента файлов (кроме DEVLOG/README из «хвостов»).
- Не «восстанавливать» файлы, чей контент совпадает с HEAD, — сначала доказать повреждение.
- Не трогать `backend/`, `frontend/` содержательно.

## Шаг 0 — бэкап (до любых других действий)

1. `git bundle create ../ai-agents-backup-$(date +%Y%m%d).bundle --all` — слепок всей истории вне рабочей папки.
2. Push `main` в `origin` — владелец, запуская этот промпт, тем самым даёт разрешение на push (обычный, НЕ force). Если push падает — доложить и продолжить без него.

## Шаг 1 — диагностика (read-only)

1. `git commit-graph verify` — подтвердить, что `improper chunk offset` от commit-graph.
2. `git update-index --refresh; git status --porcelain` — сколько «модификаций» испарится после обновления stat-кэша.
3. Для каждого файла, оставшегося modified: `git diff -- <файл>` и **прочитать сам файл**. Классифицировать: (а) контент идентичен HEAD (ложная модификация), (б) файл реально обрезан/содержит NUL-байты, (в) осмысленная правка. Ожидание по проверке Cowork: всё попадёт в (а); категория (в) маловероятна — все легитимные правки прошлого прохода закоммичены в `54b8daf`.
4. Разобраться с `prompts/_done/*` (staged D + untracked): `git ls-files --stage prompts/_done/` против содержимого папки; понять, счётся ли содержимое untracked-файлов с версиями в HEAD (`git diff --no-index`).
5. `git fsck --full` — состояние объектов.

## Шаг 2 — доложить владельцу

Промежуточный отчёт: таблица файлов по категориям (а)/(б)/(в), вердикт по fsck и commit-graph. **Если найдётся хоть один файл категории (б) или (в) — остановиться и ждать решения владельца.**

## Шаг 3 — лечение (только после чистого шага 2)

1. Commit-graph: удалить `.git/objects/info/commit-graph` (и `.git/objects/info/commit-graphs/`, если есть), затем `git commit-graph write --reachable`; перепроверить, что ошибка ушла.
2. Индекс `prompts/_done/`: если содержимое на диске идентично HEAD — `git restore --staged prompts/ && git status` (staged-удаления сняты, файлы снова tracked-чистые). Если после этого остаётся мусор — доложить, не изобретать.
3. Ложные модификации после `update-index --refresh` должны исчезнуть сами; если нет — `git status` разбирать пофайлово, не «лечить» скопом.
