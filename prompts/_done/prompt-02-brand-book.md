## Контекст

`09_Design_System/Visual_Reference.md` сейчас описывает **концепцию** Command Center (карта-как-игра, уровни масштаба, контуры целей, советники) — это ценно и остаётся. Но там НЕТ дизайн-системы: токенов, типографики, каталога компонентов, состояний, движения. При этом в коде визуальная система уже проработана:

- `frontend/src/index.css` — токены (`:root`), базовые стили, анимации, reduced-motion.
- `frontend/src/os/os.css` — полный каталог компонентов двух экранов (панель управления `.os-panel` + карточка цели `.os-goal`).
- `frontend/src/os/` — `CommandPanel.tsx`, `GoalCard.tsx`, `ProcessMap.tsx`, `data.ts`, `goals.ts`.

Задача — извлечь эту систему из кода и зафиксировать бренд-буком, чтобы все следующие блоки UI ссылались на него, а не переизобретали стиль.

## Структура результата

Дополнить `Visual_Reference.md` **второй частью «Дизайн-система (as-built)»**, сохранив существующую часть про концепцию как «Часть I. Концепция». Не удалять концептуальные разделы. В шапке дать оглавление двух частей и пометку, что Часть II извлечена из кода и обязана ему соответствовать.

Часть II наполнить разделами ниже. Каждое значение — брать из кода дословно (hex, px, имена переменных, имена классов), не округлять и не выдумывать.

### D1. Идентичность и тон
- Название эстетики в коде — **Terminal Strategist** (комментарий в `index.css`); продукт в коде подписан **Вектор·OS**. Зафиксировать как рабочие имена визуального языка.
- Тон: стратегический командный центр, тёмная тема, моно-акценты «терминала», сдержанная геймификация (без конфетти/бейджей) — согласовать с Management_Model §8 и Visual_Reference Частью I.
- **Открытый вопрос (не решать догадкой):** в репозитории несколько имён продукта — «AI Company Simulator» (PRD), «Decision Center» (README), «Вектор·OS» / «Command Center» (код/визуал). Вынести в раздел «Открытые вопросы» бренд-бука как нерешённое именование, не выбирая победителя самовольно.

### D2. Цветовые токены (из `index.css :root`)
Задокументировать таблицей: переменная → hex/rgba → семантика по факту использования в `os.css`. Как минимум:
- Фон/поверхности: `--bg #0a0a0b`, `--bar #0e0e10`, `--sf #141416`, `--sf2 #1b1b1e`, `--ln rgba(255,255,255,.08)`.
- Чернила: `--ink #e4e3df` и шкала `--i55 / --i40 / --i35` (прозрачности).
- Семантические: `--gr #8fd14f` (здоровье/выполнено/подтверждение/AI-ok), `--op #e8c04a` (внимание/активно/текущий шаг/перегруз-warn), `--rk #e8544a` (риск/перегруз/дефицит).
- Указать, где в коде цвета захардкожены мимо токенов (например `#101012`, `#161618`, `#1e2117` в `os.css`) — зафиксировать как **долг консистентности** в «Открытых вопросах», НЕ править в этом проходе.

### D3. Типографика
- Семейства: `--sans` (IBM Plex Sans …), `--mono` (IBM Plex Mono …). Правило: моно — для чисел, ID, метрик, кодов, меток состояний; sans — для контента.
- Собрать фактическую шкалу из `os.css` (размеры/начертания реально встречающиеся: напр. 22/19/15/13.5/13/12.5/12/11/10/9.5/9 px с весами 400/500/600/700). Оформить как таблицу «роль → font-shorthand → где применяется».

### D4. Сетка, отступы, форма
- Радиус скругления (доминирует `4px`; отметить исключения — круги аватаров/точек).
- Толщина линий, бордеры (`--ln`, пунктиры для «черновика»/«следующего»).
- Ключевые раскладки: панель управления (top 56px, rail 178px, stage, advisors 352px, bottom 44px); карточка цели (`.wrap` max 1480px, `.cols` 1fr/350px, чейн-полоса из 5 колонок). Брать размеры из `os.css`.

### D5. Каталог компонентов (из `os.css`)
Для каждого семейства: имя-класс, назначение, состояния, где рендерится. Минимум:
- **Карта целей / узлы:** `.gcard` (+ `.g-done/.g-warn/.g-risk/.g-next`), раскрытая цель `.gx`, «туман войны» `.fog`, BI-вкладка `.bitab`.
- **Карточка цели:** заголовок `.hd` + KPI-плитки `.k`; чейн-полоса `.chain/.ch` (`.prev/.cur/.next`).
- **Ресурсы:** карточки `.rc` (свои/roll-up переключатель `.rtog`), строка юнита `.rrow` + `.rload` + `.rcost`, единый пикер людей и AI `.rpool`, сигнал перегруза `.ovl` (+`.warn`).
- **Этапы-цели:** `.wstage` (кликабельные, `role=button`, синхрон с картой), lifecycle-чипы `.chip.lc-*`.
- **Карта процесса:** `.pmap` — узлы `.pm-node` (`.pm-passed/.pm-next/.hot`), рёбра `.pm-edge` (`.pm-passed/.pm-now/.pm-ready/.pm-next`), гейты `.pm-gate`, условия `.pm-cond`, стрелки `.pm-arr`, легенда `.pm-legend`. Явно связать со split/join/condition из Management_Model §1.2.
- **Советники:** панель `.adv`, слоты `.slot` (+`.on/.add`), чат `.msg` (`.ai/.me`), быстрые кнопки `.quick`, ввод `.inp`.
- **Проверки увязки:** `.checks/.check` (`.ok/.wr/.er`) — связать с advisorChecks и Management_Model §3.
- **Бейджи/чипы/аватары:** `.bdg` (`.b-act/.b-sec`), `.chip` (`.ai/.miss`), `.ava`, точки `.dot`.

### D6. Семантика состояний (свести воедино)
Единая таблица «состояние → визуальный код → смысл», связав с каноном:
- Lifecycle (Management_Model §3.3, PRD §14.3): draft/active/review/archived → чипы `.lc-*`.
- Поток процесса: passed/now/next → классы `pm-*`, `flowState()` в `goals.ts`.
- Определённость: **туман войны** (`.fog`) и **бэклог** — визуальные сигналы «надо доработать» (Management_Model §3.3).
- Перегруз ресурсов (§41.1): `.ovl`/`.ovl.warn`, логика `findOverloads()` в `goals.ts`.

### D7. Движение
- `os-pulse` (2.4s, активный/текущий узел), `os-flow` (1.6s, поток по рёбрам/связям). Где применяются.
- Обязательный `@media (prefers-reduced-motion: reduce)` — все анимации off. Зафиксировать как правило доступности.

### D8. Доступность
- Фокус-обводки `:focus-visible` (2px `--gr`) — уже в коде, задокументировать как стандарт.
- Reduced-motion (D7). Контраст на тёмном фоне. Требование локализации: весь UI-текст на русском (CONTRIBUTING «Localization»).

### D9. Использование бренд-бука
Короткий раздел «как применять»: новые экраны берут токены из `index.css`, переиспользуют классы `os.css`, следуют семантике D6; отклонения — через ADR (`docs/adr/`).
