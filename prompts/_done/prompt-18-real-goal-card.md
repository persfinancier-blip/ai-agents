# Промпт №18: Шаг Ф2 — карточка цели на реальных данных (чтение)
### `feat/frontend-real-goal-card`, `feat: real goal card view (read-only)`

> **Кому:** Claude Code. Второй фронтовый срез: клик по реальному узлу карты перестаёт быть заглушкой.
> **Модель/режим:** Sonnet, обычный режим. API готов (`GET /goals/{id}`, `GET /goals/{id}/subtree`), новых бэкенд-возможностей не требуется.
> **Канон:** `docs/full-vision/09_Design_System/Visual_Reference.md` Часть II — бренд-бук (классы `.os-goal`, чипы `.lc-*`, D6 семантика состояний; новых стилей не изобретать); `docs/full-vision/02_Product/Management_Model.md` §9 (fog/defined); контракт — `frontend/src/types.ts` (`GoalRead`), клиент — `frontend/src/api.ts` (уже есть `getGoal`/`getGoalSubtree`).
> **Предусловие:** промпт №17 выполнен (карта на реальном API). Если `git status` вдруг покажет «массовые модификации» без правок — это протухший кэш, НЕ повреждение: см. запись `chore/repo-integrity-recovery` в `docs/DEVLOG.md`, не «лечи» скопом.

## Scope

**Делаем:**
1. Реальные узлы карты (`RealGoalMap` в `CommandPanel.tsx`) становятся кликабельными: клик открывает карточку цели на реальных данных.
2. Новый компонент `frontend/src/os/RealGoalCard.tsx` — рядом с демо-`GoalCard.tsx`, НЕ вместо него: демо-карта продолжает открывать демо-карточку, реальная — реальную.
3. Наполнение карточки — только то, что есть в `GoalRead`:
   - топ-бар: имя, чип lifecycle (`.chip.lc-*` по `lifecycle_stage`), бейдж «туман»/«определена» (по `definiteness`, приглушённый тон для тумана), «назад к карте»;
   - шапка `.hd`: имя, `description` (если есть), владелец («отв. —» при пустом), `role_label`;
   - KPI-плитки `.kpis > .k`: имя KPI, `target` + `unit`; у KPI без таргета — «—» (это и есть туман-кейс); у композитных (`computed_value != null`) — вычисленное значение с пометкой «расчёт»; подпись плиток — «ПЛАН» (факта не существует до ресурсных блоков, ADR-0005);
   - навигация по структуре: строка родителя (`parent_id` → `getGoal`, клик — переход вверх) и список прямых детей из `getGoalSubtree` как строки `.wstage`-типа (имя + владелец + определённость, клик — переход вглубь). Это drill-down по `parent_id`-дереву, путь держит `App.tsx` (`goalPath`), как у демо-карточки.
4. Состояния: загрузка / ошибка («Не удалось загрузить цель») / данные. 404 (цель удалили) → сообщение + возврат на карту.

**НЕ делаем (сознательно, не домысливать):**
- **Никакого редактирования** — create/patch/delete из UI не подключаем (следующий срез).
- **Ресурсы, юниты, загрузка, стоимость, перегрузы** — этих данных на бэке нет; секции ресурсов в реальной карточке не рендерить вовсе (не выдумывать и не переносить демо-значения).
- **Карту процесса (BPMN, `ProcessMap`) не рендерить** — переходов между целями на бэке нет.
- **Не выдумывать прогресс** — ни процентов, ни баров выполнения; KPI показывают план.
- **Связи KPI→KPI и циклы** (`/kpi-links`, `/kpi-links/cycles`) — API есть, но в карточку не тащить: визуализация связей — открытый вопрос ADR-0004, отдельный срез.
- **Сайдбар советника** в реальной карточке не включать (советник по целям — отдельная веха; демо-чек-лист `advisorChecks` не переносить).
- `GoalCard.tsx`, `data.ts`, `goals.ts` — не трогать. Backend — не трогать вообще.

## Ограничения

- UI-текст только на русском (CONTRIBUTING «Localization»).
- Стили — только существующие токены/классы бренд-бука (`index.css`, `os.css`); новый класс — лишь если ни одно семейство D5 не подходит, в стиле существующих; расхождения фиксировать в «Открытых вопросах» Части II.
- Windows-среда: PowerShell 5.1 — команды без `&&`, по одной; в venv битые лаунчеры — только `python -m X`.
- После merge в `main` — `git push origin main` (правило в `.claude/rules/commits.md`).
