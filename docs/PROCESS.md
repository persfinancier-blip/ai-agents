# PROCESS — протокол цикла разработки ai-agents

Единый источник правды о том, **как здесь работают**, а не «что построено» (это PRD/Management_Model) и не «что сделано» (это DEVLOG/BACKLOG). Цель — онбординг новой Cowork-сессии, сменного исполнителя или второго разработчика по одному этому файлу. Источники: [`COWORK.md`](../COWORK.md), [`CLAUDE.md`](../CLAUDE.md), [`.claude/rules/`](../.claude/rules/), [`prompts/README.md`](../prompts/README.md).

## 1. Роли

| Роль | Что делает | Что НЕ делает |
|---|---|---|
| **Владелец** | Желания, продуктовые решения, вето, подтверждение модели выше Sonnet | Не пишет код и промпты сам |
| **Cowork-архитектор** | Пишет промпты в `prompts/`, канон (ADR/Management_Model), пост-merge верификация по факту | Не коммитит, не создаёт ветки/PR, не пишет в `backend/`/`frontend/` |
| **Исполнитель Claude Code** | Единственный, кто пишет в git: код, коммиты, PR, merge. Облачный воркер (GitHub Actions) или локальный pass | — |
| **Дозор (watchdog)** | Мониторинг воркера, доклады владельцу об исходе | Не пишет в репозиторий |
| **Субагенты-ревьюеры** | `code-reviewer`, `spec-guardian`, `doc-keeper`, `test-runner` — on-demand по запросу или по условию (диф ~150+ строк, канон/модель данных, доки, DoD-проверка) | Не запускаются «всегда» (правило отменено 2026-07-14) |

Один пишущий в git агент — Claude Code; это инвариант (см. §7).

## 2. Жизненный цикл задачи (основной путь)

```
желание владельца
   → Cowork пишет prompts/prompt-*.md (не коммитит)
   → watcher (scripts/dispatch-tasks.ps1, Task Scheduler, ONLOGON) видит файл
   → отправляет как ветку task/<slug>-<timestamp>
   → push будит job claude-task-push (.github/workflows/claude.yml)
   → облачный воркер выполняет промпт, коммитит, открывает PR
   → DoD-гейт считается ВНУТРИ job (zone-scoped по git diff origin/main...HEAD)
        зелёный → auto-merge (gh pr merge --merge --delete-branch)
        красный → PR остаётся открытым, коммент 🔴 «Проверки не прошли: …»
   → Cowork делает пост-merge верификацию (читает файлы, не верит докладу)
   → откат при проблеме = revert PR (не силовые операции)
```

Правила, применяющиеся на каждом шаге, — в `.claude/rules/commits.md` и `.claude/rules/github-automation.md`. DoD-гейт не делегируется CI: PR от `GITHUB_TOKEN` не запускает `ci.yml` (анти-рекурсия GitHub) — гейт считается как обычный shell-шаг внутри `claude-task-push`.

Одна задача = один PR = один vertical slice (эндпоинт + сервис + миграция при наличии + тесты + фронт при необходимости); сквозные правки — отдельный `chore:` PR.

## 3. Запасные пути (когда основной недоступен или неприменим)

- **Локальный pass Claude Code** — обязателен для правок `.github/workflows/**` (`GITHUB_TOKEN` не может пушить workflow-изменения — «refusing to allow a GitHub App to create or update workflow … without `workflows` permission», run 29559783511) и когда машина владельца выключена (watcher не работает).
- **Issue с меткой `ai-task`** — Cowork может создать задачу как GitHub issue тем же форматом (Scope/Constraints/DoD) через GitHub MCP вместо/вместе с файлом-промптом; файл всё равно кладётся в `prompts/` как архивная копия.
- **Комментарии `@claude <note>`** на issue/PR — единственный canonical способ доработки; никаких bot-to-bot циклов и автономных Claude↔Claude ревью.
- Пути `claude-issue` и `claude-comment`, а также любой вручную открытый PR — merge только по явному слову владельца (auto-merge из §2 действует **только** для `claude-task-push`).

## 4. Формат промпта

Каждый промпт в `prompts/` содержит: заголовок (для кого, ветка, тип коммита, ссылки на канон) → **Scope** (что делаем) → **явное NOT in scope** (что не делаем — держит срез от разрастания) → тело (модель/схема/сервис/роут/миграция/тесты, по образцу существующего кода) → **Constraints** (что не трогать) → **Definition of Done** (чеклист: тесты зелёные, линт/типы, DEVLOG, один коммит, merged в `main`).

Именование:
- `prompt-NN-краткое-имя.md` — продуктовые промпты (последовательная нумерация).
- `prompt-ops-NN-краткое-имя.md` — инфраструктура/сам пайплайн Cowork↔Claude Code (независимая нумерация).

Оба архивируются в `prompts/_done/` одинаково — Claude Code коммитит перемещённый файл вместе с остальной работой в том же PR (без отдельного пост-merge переноса). `prompts/_done/` — не свалка, а образец стиля; примеры: `prompt-ops-01…06`, продуктовые `prompt-10…40`.

**Тела промптов владельцу в чат не показываются** — ни при диспетчеризации, ни в докладах, ни «для справки». Владелец получает одну строку на диспетчеризацию (имя промпта + суть в нескольких словах) и результат от дозора.

## 5. Защита от сбоев

- Каждый job в `claude.yml` — `timeout-minutes: 30`.
- Мёртвый/упавший run: `claude-issue`/`claude-comment` → коммент на issue; `claude-task-push` → issue с меткой `worker-failure` и ссылкой на run, создаётся token-free шагом `github-script`. `worker-failure` никогда не запускает воркер повторно (не самозацикливается).
- Дозор — окно мониторинга 9:00–01:00, тихий выход без новостей, если нечего доложить.
- Известные ограничения GitHub: анти-рекурсия `GITHUB_TOKEN` (PR/merge от токена не запускают `ci.yml` и последующие workflow) — отсюда гейт внутри job (§2); запрет пуша в `workflows` без permission `workflows` — отсюда обязательность локального pass для `.github/workflows/**` (§3).
- Если branch-protection блокирует merge боту — job логирует точную причину, PR описывает её владельцу; настройки репозитория агент не меняет сам.

## 6. Память и состояние

| Файл | Что в нём | Живёт |
|---|---|---|
| `docs/DEVLOG.md` | Журнал проходов: что сделано, дальше, открытые вопросы | Хронологически, не перезаписывается |
| `BACKLOG.md` | Очередь задач | Обновляется каждым проходом |
| `docs/adr/` | Решения по архитектуре/модели («почему так»), включая нерешённые вопросы | Одно ADR = одно решение/направление |
| `/handoff` | Перенос контекста между Cowork-сессиями (см. §8, «Сессионная гигиена») | Генерируется по команде, не хранится |
| agent-memory (`.claude/…`/память Claude) | Профиль владельца, обратная связь по подходу, контекст проекта | Между сессиями Claude Code |

Всё остальное восстанавливается из репозитория автоматически: код, `git log`, текущее состояние доков — Cowork и Claude Code не полагаются на память там, где можно перечитать файл.

## 7. Инварианты (нарушать нельзя)

1. **Один пишущий в git агент** — Claude Code. Cowork читает через GitHub MCP (read-only), не коммитит и не пушит.
2. **Канон прежде кода** — новая механика сначала фиксируется в `Management_Model.md` + ADR, потом реализуется.
3. **Вертикальные срезы** — один PR = один слой end-to-end, не «слой ради слоя».
4. **Token economy**: модель — Sonnet (Opus/выше только по явному слову владельца, зафиксированному как подтверждение); минимальный скоуп чтения (файл → папка → модуль, не весь репозиторий); короткие доклады (выводы и действия, без содержимого файлов/диффов/цитат промптов).
5. **Прямой push в `main` — только владелец.** Хук `protect-main` блокирует прямые правки агентов; агенты всегда идут через PR.
6. **Секреты никогда не в файлах** — LLM-ключи и токены через переменные окружения; локально `LLM_PROVIDER=stub` не требует ключей вовсе.

## 8. Состояния задачи (мини-FSM)

```
Idle → Prompt Ready → Dispatched → Executing → Gate → Merged / Blocked → Verified → (Reverted)
```

| Переход | Кто совершает |
|---|---|
| Idle → Prompt Ready | Cowork пишет и сохраняет файл в `prompts/` |
| Prompt Ready → Dispatched | watcher (событийно) или ручной kickoff (fallback) |
| Dispatched → Executing | GitHub Actions job (`claude-task-push`/`claude-issue`/`claude-comment`) |
| Executing → Gate | Claude Code завершает работу, открывает PR |
| Gate → Merged | DoD-гейт зелёный → auto-merge (только `claude-task-push`) или явное слово владельца (остальные пути) |
| Gate → Blocked | DoD-гейт красный → PR открыт, коммент 🔴 |
| Blocked → Executing | доработка по `@claude <note>` на issue/PR |
| Merged → Verified | Cowork читает результат по факту |
| Verified → Reverted | владелец/Cowork при обнаруженной проблеме — через revert PR |

## Сессионная гигиена (Cowork)

Сессия = одна заявленная цель + проверяемый критерий «готово». Выход из сессии → `/handoff` → `/clear` → новая сессия — при достижении цели или при уходе контекста в сторону. Триггер `/handoff` привязан к моменту «пост-merge верификация подтвердила веху» — в этом же сообщении предлагается `/handoff`, новая работа в закрытой сессии не начинается. Детали — `COWORK.md` → «Session hygiene».

## Открытые вопросы

- Владелец кодифицировал «Ф8» роадмап и веху токен-экономии 2026-07-14 отдельными решениями, но общего документа-протокола (этого файла) до сих пор не существовало — часть правил жила «в традиции» разрозненно по `COWORK.md`/`CLAUDE.md`/`.claude/rules/`. Этот пасс их только собирает, ничего не меняя; сверка формулировок с источниками рекомендуется после первого использования новым участником.
- `.claude/rules/` содержит `backend.md`/`frontend.md`/`docs.md`/`commits.md`/`github-automation.md`, но не отдельный `process.md` — этот файл частично их дублирует по смыслу (roles/lifecycle не жили ни в одном rules-файле явно). Стоит ли выносить пересечение в `.claude/rules/process.md` как zone-file — решение владельца, не принято этим проходом.
- `prompts/README.md` описывает «Активные» промпты как пустой список, но одновременно перечисляет полный список выполненных N-промптов в этом же разделе — вероятная опечатка структуры файла (список должен быть под «История», не под «Активные»); не исправлялась, т.к. вне scope этого прохода.
