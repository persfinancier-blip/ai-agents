# Workspace Platform — рабочие пространства сущностей

**Версия:** 1.0
**Статус:** Architecture Direction (направление, НЕ активный спек текущего среза)
**Связанные документы:** `02_Product/PRD.md`, `09_Design_System/Visual_Reference.md`, Entity_Platform.md, `07_Business_Capabilities/Business_Capability_Map.md`, `08_Enterprise_Analytics/Dashboard_Builder.md`

---

# 1. Назначение

Workspace Platform — универсальный пользовательский слой над Entity Platform.

Документ отвечает на вопрос: **что видит пользователь, когда открывает сущность**. Ответ: не форму и не список, а операционное пространство этой сущности.

Документ сводит в одну модель три ранее разрозненные концепции:

- блоки-районы Command Center (`Visual_Reference.md`, раздел 6);
- Workspace конструктора дашбордов (`Dashboard_Builder.md`, раздел 4);
- Decision Canvas (`Decision_Center.md`).

Все три — частные случаи одного механизма, описанного здесь.

---

# 2. Основной принцип

**Every Entity Has Workspace** — каждая сущность имеет рабочее пространство.

Это второй фундаментальный принцип платформы, продолжающий первый (`Entity_Platform.md`, раздел 2). Вместе они образуют цепочку:

```
Everything is Entity
(всё есть Сущность)

↓

Every Entity Has Workspace
(каждая Сущность имеет рабочее пространство)

↓

Every Workspace is Composed of Capabilities
(пространство собирается из возможностей)

↓

Every Capability is Implemented Through Entities
(каждая возможность реализована через Сущности и сервисы)
```

Пространство не создаётся вручную — оно **автоматически существует** у любой сущности, потому что является проекцией самой сущности: её свойств, связей, памяти, знаний, метрик и событий (универсальная модель Entity, `Entity_Platform.md`, раздел 5).

---

# 3. Позиционирование: карта остаётся главной

Принцип НЕ отменяет канон интерфейса (`Visual_Reference.md`, раздел 2): **главный экран системы — карта целей**.

Разделение ответственности:

- **Карта целей** — навигация и обзор: где мы, что происходит, куда смотреть. Уровни масштаба (компания → отдел → сотрудник → агент) — перемещение по карте.
- **Рабочее пространство** — «внутренность» узла: зум в любую сущность на карте открывает её пространство.

То есть пользователь не выбирает пункт меню «Проекты» — он проваливается в узел проекта на карте и оказывается внутри пространства этого проекта. Формы вторичны и открываются из контекста пространства, как раньше — из контекста карты.

```
Карта целей (главный экран)

↓ зум в узел

Рабочее пространство сущности

↓ действие в пространстве

Форма / виджет / диалог с агентом
```

---

# 4. Что такое рабочее пространство

Операционная среда конкретной сущности. Не папка с документами, не карточка со свойствами.

Пространство сущности объединяет:

- **обзор** — состояние, Health Score, зрелость, ответственный;
- **цели и KPI** — участок карты целей, относящийся к сущности;
- **решения** — привязанные Decision и их стадии;
- **юниты** — люди и AI-агенты, работающие в контуре сущности;
- **знания** — документы, SOP, база знаний сущности (Entity Knowledge);
- **память** — история, события, принятые решения, ошибки, достижения (Entity Memory);
- **аналитику** — дашборды и отчёты по сущности;
- **коммуникации** — обсуждения в контексте сущности;
- **AI-контекст** — диалог с агентами, которые видят всю сущность целиком (AI Context, `Entity_Platform.md`, раздел 17).

Состав конкретного пространства определяется его возможностями (раздел 5), а не фиксированным списком вкладок.

---

# 5. Пространство собирается из возможностей

Пространство — контейнер. Наполнение — **Capabilities** из `Business_Capability_Map.md`.

Правила:

- **Не вкладки, а возможности.** Пространство не имеет фиксированного набора экранов; оно подключает возможности по мере потребности («нужен отчёт» → подключается Analytics & Reporting).
- **Активация по зрелости.** Возможность подключается, когда сущность до неё дозрела, — тот же принцип, что активация блоков-районов (`Visual_Reference.md`, раздел 6: «как строительство здания в стратегии»). Рекомендацию «пора подключить» даёт Советник.
- **Блоки-районы — частный случай.** Универсальные блоки управления Command Center — это возможности пространства компании/подразделения. Настоящий документ обобщает механизм на любой тип сущности.
- **Пространство эволюционирует вместе с сущностью.** Рост зрелости (Initial → … → Autonomous, `Entity_Platform.md`, раздел 11) открывает новые возможности; накопление памяти и знаний наполняет существующие. Пространство проекта в день создания и через полгода — разные по насыщенности среды при одном механизме.

## Стартовый состав по типам сущностей (ориентир, не догма)

| Сущность | Ядро пространства | Подключается по зрелости |
|---|---|---|
| Компания / Подразделение | карта целей участка, блоки-районы, юниты | финансы, риски, университет |
| Проект | цели, задачи, команда, бюджет | риски проекта, знания проекта |
| Сотрудник (Human) | личная карта, задачи, его агенты, профиль компетенций | план развития, история оценок |
| AI-сотрудник | настройка, обучение, метрики, автономность | датасеты, аттестации |
| Клиент | сделки, договоры, коммуникации | история, аналитика LTV |
| Документ / SOP | содержимое, версии, связи | обсуждения, AI-анализ |
| Решение (Decision) | Decision Canvas: стадии, борд, источники | история изменений, симуляция |

---

# 6. Виджеты

Внутренняя компоновка возможности в пространстве — виджеты.

Механизм уже описан в `Dashboard_Builder.md` (разделы 4–5): панели, drag & drop, шаблоны, AI Assistant. Уточнение терминологии: **Workspace из `Dashboard_Builder.md` — это реализация рабочего пространства**, а дашборд — одна из возможностей внутри него, не самостоятельная концепция.

Владелец пространства может пересобирать компоновку под себя; разумная компоновка по умолчанию генерируется из типа сущности и подключённых возможностей.

# 7. Маппинг на Entity Platform

Новых моделей данных не появляется (`Entity_Platform.md`, раздел 20).

| Элемент | Реализация |
|---|---|
| Пространство сущности | проекция самой Entity: свойства + связи + память + знания + метрики |
| Сохранённая конфигурация пространства | Entity типа Workspace (уже в классификации, `Entity_Platform.md`, раздел 6) |
| Возможность (Capability) | домен из `Business_Capability_Map.md`, реализованный через Entities и сервисы |
| Виджет | Entity типа Widget (классификация Analytics) |
| Подключение возможности | событие + отношение Entity Graph |

Пространство «живое» не потому, что это отдельный механизм, а потому, что живая сама сущность (принципы Everything Evolves / Everything Remembers, `Entity_Platform.md`, раздел 4).

# 8. Терминология

Единый термин: **рабочее пространство сущности** (в коде и API — Workspace).

Отклонённые синонимы — не употреблять в доках и UI: Operational Workspace, Enterprise Workspace, Living Workspace, Workspace OS. Пользовательский UI-текст — на русском (CONTRIBUTING, «Localization»).

---

# 9. Не-цели

- **Не замена карты целей.** Главный экран — карта (`Visual_Reference.md`, раздел 2); пространство — деталь узла.
- **Не текущий срез.** Реализация начинается не раньше, чем ядро (цели / связи / ресурсные блоки) заработает и карта целей будет привязана к реальному API (см. BACKLOG).
- **Не «Enterprise Browser».** Встраивание внешних сервисов (GitHub / Docs / Telegram) в пространство — отдельная идея с низким приоритетом (BACKLOG, «скептически»), сюда не входит.
- **Не новый слой данных.** Любая попытка завести под пространства собственные модели вне Entity Platform — нарушение канона.

---

# 10. Контур MVP (когда дойдёт очередь)

Поэтапно, каждый шаг — работающий:

1. **Пространство цели**: деталь узла карты целей как первое пространство — обзор, KPI, связи, привязанные решения. Данные уже есть (Goal + Kpi + kpi_link + Decision).
2. **Пространство подразделения**: участок карты + юниты + блоки-районы (проекция уже описанного в `Visual_Reference.md`, раздел 6).
3. **Пространство сотрудника и агента**: личная карта, профиль компетенций, метрики агентов (`Visual_Reference.md`, разделы 5, 7).
4. **Конструктор**: пересборка компоновки, подключение возможностей, рекомендации Советника.

---

# 11. Назначение документа

Workspace Platform определяет универсальный пользовательский слой Enterprise Operating System: каждая сущность автоматически имеет рабочее пространство — свою операционную среду, собираемую из возможностей Business Capability Map и наполняемую данными самой сущности через Entity Platform. Карта целей остаётся главным экраном и навигацией; пространство — то, что открывается при зуме в узел. Блоки-районы Command Center, Workspace конструктора дашбордов и Decision Canvas — частные случаи этого механизма.
