## 1. Назначение

Integration Hub является единой интеграционной шиной (Enterprise Service Bus) Enterprise Operating System.

Он обеспечивает:

- бесшовное взаимодействие между всеми внутренними сервисами платформы;
- интеграцию с внешними системами (SaaS, корпоративные приложения, API партнёров);
- единый протокол обмена событиями и данными;
- трансформацию и маршрутизацию сообщений;
- безопасный и аудируемый обмен информацией;
- поддержку синхронных и асинхронных паттернов;
- мониторинг и управление интеграционными потоками.

Integration Hub рассматривается как самостоятельная Сущность (Entity) платформы и подчиняется всем принципам Entity Platform.

---

## 2. Архитектурные принципы

### 2.1. Everything is Integration
Каждое взаимодействие между сущностями (внутренними или внешними) является интеграционным событием и должно проходить через Hub.

### 2.2. Loose Coupling
Сервисы не должны знать о внутренней структуре друг друга. Все взаимодействия строятся через контракты (схемы, API, события).

### 2.3. Event-Driven First
Hub является центральной точкой публикации и подписки на события. Все изменения сущностей генерируют события, которые могут быть обработаны другими сервисами.

### 2.4. Polyglot Integration
Hub поддерживает разнообразные протоколы (REST, gRPC, GraphQL, WebSocket, AMQP, MQTT, файловые обмены) и форматы данных (JSON, XML, Protobuf, Avro, CSV).

### 2.5. Security by Design
Все интеграционные потоки проходят аутентификацию, авторизацию, шифрование и аудит.

### 2.6. Observability
Каждый интеграционный вызов трассируется, логируется и предоставляет метрики для мониторинга и анализа.

### 2.7. Autonomous Evolution
Hub должен позволять добавлять новые интеграции без изменения существующих сервисов (плагинная архитектура).

---

## 3. Место в общей архитектуре

Integration Hub является **центральной шиной**, через которую проходят все коммуникации между компонентами Enterprise Operating System.


Hub является **единственной точкой входа** для внешних интеграций и **основным каналом** для внутренней коммуникации (хотя внутренние сервисы могут также общаться напрямую через gRPC/HTTP в высоконагруженных сценариях, но по возможности должны использовать события Hub).

## 5. Компоненты Integration Hub

### 5.1. API Gateway
Единая точка входа для всех синхронных запросов (REST, GraphQL, gRPC).

Функции:
- маршрутизация запросов к соответствующим микросервисам;
- аутентификация и авторизация (JWT, OAuth2, API-ключи);
- rate limiting и throttling;
- кэширование ответов;
- валидация запросов (схемы JSON, Protobuf);
- трансформация ответов (например, из внутреннего формата во внешний).

### 5.2. Event Bus
Центральная шина событий (на основе Apache Kafka / RabbitMQ).

Функции:
- публикация событий сущностей (EntityCreated, EntityUpdated, DecisionMade и т.д.);
- доставка событий подписчикам с гарантией (at-least-once, exactly-once);
- управление очередями и топиками;
- replay событий для восстановления состояния;
- поддержка событийного sourcing (Event Sourcing) при необходимости.

### 5.3. Connectors
Предварительно настроенные адаптеры для популярных внешних систем.

Каждый Connector является самостоятельным модулем, который:
- реализует специфический протокол и аутентификацию;
- преобразует внешние форматы данных в единую модель сущностей;
- обрабатывает ошибки и повторы;
- предоставляет метрики и логи.

**Базовые коннекторы для MVP:**
- Google Workspace (Docs, Sheets, Drive, Calendar)
- Microsoft 365 (Outlook, Excel, SharePoint)
- Slack, Telegram, Email (SMTP/IMAP)
- GitHub, Jira
- CRM (HubSpot, Salesforce – через REST)
- ERP (SAP, 1С – через адаптеры)
- Базы данных (SQL, MongoDB через CDC)

### 5.4. Webhook Engine
Позволяет внешним системам подписываться на события платформы.

Функции:
- регистрация вебхуков с фильтрацией по типу события;
- доставка уведомлений на внешние URL;
- повторные попытки и экспоненциальная задержка;
- подпись запросов для верификации (HMAC).

### 5.5. MCP Server
Реализует протокол Model Context Protocol для взаимодействия AI-агентов с внешними инструментами.
- AI-агенты через MCP могут вызывать инструменты (Google Docs, базы знаний, API).
- Hub преобразует MCP-вызовы в соответствующие интеграционные действия.
- Поддерживается MCP для всех Connectors.

### 5.6. Transformation Engine
Отвечает за преобразование данных между форматами и схемами.
- Поддержка JSON, XML, Protobuf, Avro, CSV.
- Преобразование по правилам (XSLT, JSONata, пользовательские скрипты).
- Валидация входных и выходных данных.

### 5.7. Protocol Adapters
Обеспечивают поддержку различных протоколов:
- REST/HTTP
- GraphQL
- gRPC
- WebSocket
- AMQP (RabbitMQ)
- MQTT
- SFTP/FTPS
- SOAP (legacy)

### 5.8. Security Layer
- Аутентификация (OAuth2, JWT, API-ключи, Basic Auth, клиентские сертификаты).
- Авторизация (RBAC, ABAC) на уровне запросов и событий.
- Шифрование данных в транзите (TLS 1.3) и в состоянии покоя (при необходимости).
- Аудит всех интеграционных действий.
- Сканирование на угрозы (SQL-инъекции, XSS, инъекции в JSON).

### 5.9. Observability
- Логирование каждого запроса и события (структурированные логи).
- Распределённая трассировка (OpenTelemetry).
- Метрики (количество запросов, ошибки, задержки).
- Панель мониторинга состояния интеграций.
- Оповещения при сбоях и превышении порогов.

### 5.10. Registry & Catalog
Хранилище всех доступных интеграций, коннекторов, API-схем и контрактов.
- Автоматическое документирование API (OpenAPI, AsyncAPI).
- Версионирование контрактов.
- Поиск и обнаружение интеграций.

## 6. Интеграционные паттерны

Hub поддерживает следующие паттерны:

- **Request-Response** – синхронные вызовы через API Gateway.
- **Event-driven** – асинхронные события через Event Bus.
- **Command-Query Responsibility Segregation (CQRS)** – разделение команд и запросов (при необходимости).
- **Saga** – распределённые транзакции через события и компенсации.
- **Aggregator** – агрегация данных из нескольких источников.
- **Splitter** – разбиение составных сообщений.
- **Router** – маршрутизация на основе содержимого.
- **Translator** – преобразование форматов.
- **Message Queue** – очереди для надёжной доставки.
- **Publish-Subscribe** – подписка на события.

---

## 7. Взаимодействие с Entity Platform

Hub напрямую интегрирован с Entity Platform:
- Любое изменение сущности автоматически порождает событие (EntityEvent), которое публикуется в Event Bus.
- Все внешние интеграции работают с сущностями через API Gateway (CRUD, поиск, отношения).
- Connectors могут маппить внешние объекты на внутренние сущности.
- Hub использует Entity Graph для обогащения событий контекстом.

---

## 8. Взаимодействие с AI Workforce

Hub является мостом для AI-сотрудников:
- AI-агенты через MCP вызывают инструменты (Tool Calling) – эти вызовы маршрутизируются через Hub к соответствующим Connectors.
- Hub возвращает результаты выполнения инструментов агентам.
- AI-агенты могут подписываться на события (например, новые задачи) через Event Bus.
- Hub обеспечивает безопасный доступ AI к внешним системам с соблюдением политик.

---

## 9. Взаимодействие с Decision Center

Decision Center использует Hub для:
- Получения внешних данных для симуляций (рыночные данные через Connectors).
- Отправки решений в другие системы (например, создание задачи в Jira).
- Получения событий от внешних систем для запуска анализа.

---

## 10. Масштабируемость и отказоустойчивость

- API Gateway и Event Bus горизонтально масштабируются.
- Connectors запускаются как отдельные микросервисы.
- Используются очереди с гарантией доставки.
- Предусмотрены повторы и обработка ошибок.
- Реализован Circuit Breaker для защиты от сбоев внешних систем.
- Hub поддерживает деградацию и fallback сценарии.

---

## 11. Безопасность и управление доступом

- Все интеграции проходят через Security Layer.
- Конфиденциальные данные (API-ключи, пароли) хранятся в защищённом хранилище (Vault).
- Доступ к интеграциям управляется через RBAC (роли: администратор интеграций, разработчик, пользователь).
- Каждая интеграция имеет владельца и политику доступа.

## 12. MVP

Для минимально жизнеспособного продукта необходимы:

- API Gateway (базовая маршрутизация, аутентификация, rate limiting).
- Event Bus (публикация/подписка на события сущностей).
- Базовые Connectors:
  - Google Sheets (чтение/запись данных).
  - Slack / Telegram (уведомления).

- Transformation Engine (JSON <-> внутренняя модель).
- Observability (логи, базовые метрики).
- Registry (документирование API).

## 15. Критерии качества

- Время отклика API Gateway < 50 мс (p95).
- Доставка событий с задержкой < 100 мс.
- Доступность Hub >= 99.99%.
- Потеря событий < 0.001%.
- Время добавления нового Connector < 1 человеко-дня (для стандартных протоколов).

---

## 16. Технический стек (предварительный)

- API Gateway: Kong / Envoy / Traefik (или кастомный на Node.js/Go).
- Event Bus: Apache Kafka / RabbitMQ.
- Connectors: на Python (FastAPI) или Node.js (TypeScript).
- Реестр: PostgreSQL + Elasticsearch.
- Мониторинг: Prometheus + Grafana + OpenTelemetry.
- Хранилище секретов: HashiCorp Vault.

## 17. Назначение документа

Integration Hub определяет архитектуру единой интеграционной шины Enterprise Operating System. Он является обязательным элементом для обеспечения связанности всех компонентов, взаимодействия с внешним миром и соблюдения архитектурных принципов «Everything is Connected» и «Everything Generates Events».

Hub служит основой для реализации всех интеграционных потребностей платформы, обеспечивая гибкость, безопасность и наблюдаемость.
