# User Scenarios — ролевые сценарии использования

**Версия:** 1.0
**Статус:** Ориентиры приоритизации (НЕ спецификация, НЕ бэклог задач)
**Связанные документы:** `PRD.md`, `Management_Model.md`, `../09_Design_System/Visual_Reference.md`, `../04_Simulation/Decision_Center.md`, `../03_AI_Workforce/Unified_Workforce.md`, `../08_Enterprise_Analytics/Report_Builder.md`, `../../adr/0004-goal-link-types-cycles-composite-kpi.md`, `../../adr/0005-plan-vs-fact-and-resource-blocks.md`

---

# 1. Назначение

Шесть ролевых сценариев продиктованы владельцем как проверка «для кого это вообще»: если экран не отвечает ни одному сценарию ниже — это повод усомниться в приоритете, а не повод его строить. Формулировки дистиллированы из слов владельца с сохранением сути (и, где было уместно, сарказма) — это не техническое задание и не очередь задач, а компас для приоритизации Command Center и Workspace Platform.

Каждый сценарий закрывает: **что нужно роли** → **какой раздел канона это покрывает** → **какие механики уже есть в коде / какие вехи ещё нужны**.

---

# 2. Основатель

> Вектор и стратегия компании. Системные проблемы и системные успехи. Хочет иметь возможность докопаться «до винтика» — до всех этапов всех целей, если понадобится.

**Покрывает:** карта целей как главный экран + уровни масштаба (Часть I §3 `Visual_Reference.md`: компания → подразделение → сотрудник → агент) + сигналы сквозных агентов (Часть I §8).

**Механики есть / вехи:** карта целей на реальных данных и drill-down по дереву целей (`parent_id`) — есть (промпты №17–18). Зум на уровень подразделения/сотрудника/агента как отдельные экраны — не реализован, сейчас один уровень (компания). Сквозные агенты — только концепция (Часть I §8); MVP-шаг 4 `Visual_Reference.md` §11 отводит им детерминированные проверки увязки до полноценных LLM-агентов.

---

# 3. Топ-менеджер

> Системность, управляемость, минимум хаоса.

**Покрывает:** увязка внешних и внутренних целей, циклы и перегрузы графа KPI (ADR-0004), Decision Center как контур принятия решений.

**Механики есть / вехи:** граф связей KPI→KPI (`kpi_link`), детект циклов (`kpi_link_cycle`), составной KPI (`kpi_factor`) — реализованы на бэкенде (ADR-0004). Визуализация связей и циклов в UI — открытый вопрос ADR-0004, не сделана. Decision Center как рабочий контур (борд CFO/COO/CTO, синтез рекомендации) — только концепция `Decision_Center.md`, кодовый срез — M3 в `BACKLOG.md`.

---

# 4. Руководитель отдела

> «Получу ли я премию за квартал/год — и если нет, кто в этом виноват». План/факт своего узла и вклад подчинённых.

**Покрывает:** декомпозиция узла на карте (Часть I §3–§4), план vs факт KPI и ресурсные блоки как источник факта (ADR-0005).

**Механики есть / вехи:** декомпозиция узла и KPI-план в карточке цели — есть (реальная карточка, промпт №18). Факт KPI и ресурсные блоки как его источник — направление, не реализовано (ADR-0005 — Proposed).

---

# 5. Исполнитель (дизайнер)

> «Плевать на высшие материи». Сколько роликов запилить, агент помогает пилить, быстрая организация в архив и автозагрузка на каналы.

**Покрывает:** пространство сотрудника без стратегического слоя — задачи + агент-помощник + автоматизация рутины (`05_Architecture/Workspace_Platform.md` §5, стартовый состав пространства «Сотрудник»).

**Механики есть / вехи:** ничего из этого сценария не реализовано — самый дальний от текущего среза. Требует Workspace Platform (пока направление, не активный спек) и, отдельно, интеграции с внешними каналами публикации — вне текущего канона (идея «Enterprise Browser» в `BACKLOG.md` зафиксирована скептически, не как обязательство).

---

# 6. Менеджер маркетплейса

> «Я пилот, агенты — штурманы». Метрики и воронки магазина, где именно просадка; обученные агенты-специалисты (аналитик-сводки, дизайнер-инфографики под праздники, перформанс-менеджер рекламы); справочники артикулов, где характеристики не теряются при миграции между маркетплейсами, поставки, юнит-экономика каждого SKU.

**Покрывает:** ресурсные блоки «Продажи и клиенты» + «Товары и услуги» (`Visual_Reference.md` §6, ADR-0005) как первое «пространство из блоков»; специализированные обученные агенты (Часть I §7, Корпоративный университет).

**Механики есть / вехи:** ничего из ресурсных блоков не реализовано. `BACKLOG.md` уже фиксирует этот сценарий как лучшего кандидата на первую пару ресурсных блоков (Продажи + Товары) — сильный аргумент за приоритизацию этой вехи среди прочих направлений ADR-0005.

---

# 7. Бухгалтер (анти-сценарий)

> «Не трогайте меня». Сдать регламентированную отчётность вовремя и до копейки. ИИ ошибается — а проверять его работу означает делать её самому, то есть никакой экономии времени.

**Покрывает:** честно зафиксированное ограничение Unified Workforce (`03_AI_Workforce/Unified_Workforce.md`, PRD §28–29) — не для каждой роли рост автономности агента является целью. Стандартизированная безошибочная отчётность — направление `08_Enterprise_Analytics/Report_Builder.md`.

**Механики есть / вехи:** ничего не реализовано, и это осознанно. В отличие от остальных пяти сценариев, здесь не нужно «допилить» шкалу автономности (Entity Platform: Initial → … → Autonomous) — она в принципе не годится для роли, где человек постоянно остаётся контролёром, а не агент растит самостоятельность. Это ограничение, а не пробел; фиксируется, чтобы не проектировать для бухгалтера то же движение «к автономности», что для остальных ролей. Report_Builder закрывает практическую часть (регламентированная отчётность), не автономность агента.

---

# 8. Как использовать

Эти сценарии — не приоритетный список сами по себе и не заменяют `Roadmap.md` (§48 PRD) или `BACKLOG.md`. Использовать как проверку при спорах о приоритете: сценарий, для которого фича ничего не меняет, — слабый аргумент в её пользу; сценарий, который она резко разблокирует (особенно №6, отмеченный в `BACKLOG.md` как сильный кандидат), — довод для более раннего приоритета.
