# ADR-0005: План vs факт у KPI и ресурсные блоки как источник факта

**Статус:** Proposed — направление, НЕ немедленная реализация
**Дата:** 2026-07-08
**Авторы:** владелец продукта + AI-агент разработки

## Контекст (Context)

KPI (Management_Model §9.1, Шаг 2a) сейчас имеет только `target` — это **план**: «хочу прибыль 1000». У KPI нет измерения **факта** — «по факту прибыль 720». Разница план↔факт — центральный смысл увязки (Management_Model §3, «Увязка»): без факта система может подсветить структуру целей, но не может сказать, движемся ли мы к плану на самом деле.

Факт не должен вводиться вручную (за редким исключением) — он должен **тянуться из ресурсного блока**, отвечающего за соответствующую область: прибыль/выручка/аренда/депозит — из блока «Финансы»; «нанять 5 менеджеров» (факт укомплектованности) — из блока «Персонал / штатное расписание»; гудвилл-фактор «брак» — из блока «Производство»; источник фактора «отзывы» пока не определён. «Удалённость от метро» — редкий пример KPI, где факт логично вводить вручную (географический факт не меняется и не тянется ниоткуда).

Ресурсных блоков в модели пока нет вообще — ни как сущностей, ни как вехи в roadmap. Владелец продукта соотносит рабочий набор блоков (Финансы · Персонал · Продажи · Товары и услуги · Производство) со смысловой рамкой «труд / земля / капитал» (классическая триада факторов производства) — это ориентир для интуиции, не проектная структура: точные названия, границы и состав блоков — предмет будущего проектирования, не решаются этим ADR.

Это ADR фиксирует направление, чтобы идея не потерялась, а не запускает реализацию: реализация факта физически невозможна до появления ресурсных блоков (неоткуда тянуть), поэтому статус — Proposed, а не Accepted.

## Решение (Decision)

1. **У KPI появится факт как второе измерение рядом с планом.** План (`target`) остаётся как есть — факт добавляется, не заменяет ничего в текущей модели (Шаги 1–3b).
2. **Факт подключается к источнику — одному из ресурсных блоков.** Большинство фактов вычисляются/тянутся автоматически из блока; часть (редко) вводится вручную, когда естественного источника нет.
3. **Ресурсные блоки — отдельная будущая веха, предпосылка факта.** Пока блоков нет, факту неоткуда взяться — механизм факта нельзя строить раньше блоков. Примерный рабочий набор блоков (не финальный): Финансы, Персонал, Продажи, Товары и услуги, Производство.
4. **Граф связей и текущая механика Шагов 1–3b не меняются.** Связи (Management_Model §9.9–9.11) сейчас соединяют KPI по плану; факт потечёт по тем же рёбрам позже, когда появится, — это не требует переделки графа связей сегодня.

Альтернатива «ввести факт вручную для всех KPI прямо сейчас» отклонена: она даёт немедленно работающее поле, но противоречит цели («факт должен быть достоверным, тянущимся из системы учёта, а не переписываемым руками») и создаёт технический долг — поле, которое потом придётся массово переподключать к блокам.

## Последствия (Consequences)

Для будущей модели данных (когда ресурсные блоки появятся — не сейчас):

- KPI-сущность (Шаг 2a) получит поле(я) факта и ссылку на источник (какой ресурсный блок/канал его вычисляет).
- Составной KPI (Шаг 3b, Management_Model §9.12) в будущем будет агрегировать факт по тем же весам, что и план — единый механизм факторов, два измерения (план/факт) через него же.
- Появится веха «Ресурсные блоки» в roadmap — раньше неё этот ADR не реализуем.
- Ничего в Шагах 1–3b (KPI-сущность, дерево, связи, детект циклов, составной KPI) не меняется прямо сейчас — этот ADR не блокирует и не переделывает текущую работу.

## Открытые вопросы

Зафиксировано как нерешённое — не додумывать при будущей реализации без отдельного обсуждения:

1. **Точный набор и границы ресурсных блоков.** Финансы/Персонал/Продажи/Товары и услуги/Производство — рабочие названия, не финальный список; где проходят границы между блоками (например, «продажи» vs «товары и услуги») — не решено.
2. **Модель «источника» факта.** Pull из ресурсного блока (блок публикует значение) vs push (блок пишет в KPI при изменении) vs ручной ввод vs внешняя интеграция (например, бухгалтерская система) — не решено, вероятно разные KPI будут использовать разные модели.
3. **Частота обновления факта.** Реальном времени, по расписанию, по запросу при открытии карты — не решено.
4. **Источник фактора «отзывы».** Для гудвилл-фактора «отзывы» источник (какой ресурсный блок или внешняя система) пока не определён вообще.
