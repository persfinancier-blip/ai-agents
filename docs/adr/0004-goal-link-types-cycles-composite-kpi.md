# ADR-0004: Типы связей целей, направленность и циклы, составной KPI

**Статус:** Accepted (модель) — детали реализации уточняются при написании промпта Шага 3 / 3b
**Дата:** 2026-07-08
**Авторы:** владелец продукта + AI-агент разработки

## Контекст (Context)

ADR-0003 зафиксировал каркас графа связей KPI→KPI (Management_Model §9), но оставил 4 открытых вопроса: множественность связей, поведение при удалении KPI-источника, направленность/циклы графа, измерение качественных KPI. Обсуждение реализации Шага 3 (граф связей) закрыло все четыре и заодно обогатило модель: связи оказались недостаточно описать одним неразличимым «работает на» — реальные сценарии требуют различать «двигает цель», «конкурирует за ресурс» и «предусловие», а качественные цели без прямого числа нужно уметь измерять, а не оставлять вечно в тумане.

Нужно зафиксировать это в каноне ДО написания промпта реализации Шага 3, чтобы код не угадывал.

## Решение (Decision)

### Р1. Множественность и типы связей

Один KPI может ссылаться на несколько вышестоящих KPI. У связи есть **тип**, стартовый набор — три:

- **способствует** (`contributes`) — вклад в ценность вышестоящей цели (пример: ремонт → статус);
- **ограничивает** (`constrains`) — конфликт за ресурс/бюджет (пример: ремонт → прибыль как бюджет);
- **зависит-от** (`depends_on`) — предусловие/вход: цель не может двигаться без другой (пример: продажи ← отчёт себестоимости).

Новые типы добавляются ТОЛЬКО под конкретный сценарий, не влезающий в эти три — правило зафиксировано явно, чтобы не плодить типологию впрок.

### Р2. Удаление KPI-источника

KPI — источник истины для связи. При удалении KPI **связь исчезает как сущность** у всех потребителей снизу (метрика стала неактуальной). Связь каскадно удаляется вместе с любым своим концом (источник или потребитель). Никаких осиротевших ссылок, никаких запретов удаления.

**Следствие:** переход `patch_goal` с replace-all на **diff-sync** становится обязательным условием ДО появления связей — иначе патч цели молча удаляет связи вместе с пересоздаваемыми KPI (задача уже в `BACKLOG.md`; здесь фиксируется, что это предусловие Шага 3, а не независимая оптимизация).

### Р3. Направленность и циклы

Связи **разнонаправленные**; направление несёт тип («способствует» обычно вверх; «зависит-от» — в любую сторону, включая вбок/вниз). Правила «только вверх» нет.

Циклы в графе связей **не запрещаются**, а **детектируются и подсвечиваются** как риск «работа по кругу / работа ради работы».

**Разрешить или разорвать цикл может только ближайший общий структурный руководитель над всеми узлами цикла** — ближайший общий предок узлов цикла по дереву `parent_id` (Шаг 2b). Он либо подтверждает цикл как осознанную ставку (НИОКР, разведка — «прямой связи с прибылью нет, но а вдруг выстрелит»), снимая тревогу, либо разрывает. Ручное решение поверх авто-детекта — та же философия, что ручной «бэклог» (Шаг 1).

Граф связей ставит вопрос (нашёл цикл), дерево структуры назначает ответственного (кто вправе решить) — две сознательно разведённые оси модели (ADR-0003) впервые работают вместе.

### Р4. Качественные KPI → составной KPI

Качественная цель («гудвилл», «статус», «новый офис»), у которой нет прямого числа, раскладывается на **факторы-KPI с весами**; её оценка = взвешенная сумма факторов (пример: гудвилл = 0.3·отзывы + 0.3·LTV + 0.4·брак).

Составной KPI = KPI (уже сущность с Шага 2a), чьё значение собрано из дочерних KPI через веса — новой сущности не требуется. Правило тумана расширяется: узел «определён», если есть таргет **ИЛИ** есть формула из факторов (каждый фактор сам измерим).

Характер — «решение в моменте»: применяется точечно, когда конкретную качественную цель нужно разложить, а не вводится сразу для всех KPI. Отдельный **Шаг 3b**, после Шага 3 (связи).

Полное раскрытие с примерами и диаграммами — `Management_Model.md` §9.9–9.12.

### Закрывает открытые вопросы ADR-0003

| № | Вопрос ADR-0003 | Статус |
|---|---|---|
| 1 | Множественные связи одной командной цели | **Решено (Р1):** да, множественность в обе стороны (§9.10) — ограничений на число связей нет. |
| 2 | Удаление KPI-источника | **Решено (Р2):** каскадное удаление связи вместе с любым её концом. |
| 3 | Направленность и циклы графа | **Решено (Р3):** разнонаправленные связи, циклы не запрещены — детектируются и подсвечиваются; разрешает/разрывает ближайший общий структурный предок. |
| 4 | Измерение качественных KPI | **Решено (Р4):** составной KPI — факторы с весами, точечно, Шаг 3b. |

## Последствия (Consequences)

Для будущей модели данных (конкретика для реализации):

- Таблица связей KPI→KPI получает поле `link_type ∈ {contributes, constrains, depends_on}`. Направление задаётся парой `(source_kpi_id, target_kpi_id)` без ограничения «только вверх» — семантику направления определяет `link_type`, не позиция в дереве.
- Детект циклов — обход графа связей (аналогично `would_create_cycle` для структуры, но на другом графе и с другим следствием: не запрет, а флаг). «Судья» вычисляется как ближайший общий предок узлов цикла по `goal.parent_id` (переиспользует структурное дерево Шага 2b). У цикла — ручной флаг «подтверждён как ставка», не влияющий на автодетект остальных циклов.
- Составной KPI (Шаг 3b) — механизм «KPI из взвешенных факторов-KPI» на существующей сущности KPI (Шаг 2a); правило определённости расширяется (таргет ИЛИ формула), но не заменяется.
- **Diff-sync в `patch_goal`** (уже в `BACKLOG.md`) — предусловие Шага 3, не отдельная оптимизация: без него связи будут молча рваться при каждом патче цели.
- Шаг 3 сам может дробиться при написании промпта реализации (например: базовая таблица связей → типы → детект циклов → судья/флаг — отдельными вертикальными срезами); это решение оставлено на момент написания промпта, не фиксируется здесь.

## Открытые вопросы

Зафиксировано как нерешённое — не достраивать догадками при реализации Шага 3/3b:

1. **Визуализация двух осей + типов связей на карте.** `Management_Model.md` §9.8 уже фиксировал открытым вопрос совместного отображения структурной свёртки ресурсов и смысловой увязки; типы связей (способствует/ограничивает/зависит-от) добавляют third измерение к тому же вопросу. Остаётся открытым до фронт-среза.
2. **Веса и нормализация факторов составного KPI.** Как складывать разноединичные факторы (проценты? нормализация 0–1? другая шкала?) — не решено, не изобретать на Шаге 3b без отдельного обсуждения.
