# ADR-0002: Детерминированная увязка для первого среза Goal Entity

**Статус:** Принято
**Дата:** 2026-07-08
**Авторы:** владелец продукта + AI-агент разработки

## Контекст (Context)

`Management_Model.md` §3–§4 (Ф3–Ф4) описывает увязку дерева целей с ресурсным слоем: разрывы между требованиями узла и доступными ресурсами подсвечиваются, но не блокируют работу, а офисные агенты (фронт/мидл/бэк) со временем дают LLM-справки о достижимости. При фиксации канона (2026-07-07) раздел §7 оставил 7 вопросов открытыми — они блокировали проектирование модели данных для первого кодового среза.

Первый кодовый срез — **Goal Entity** (дерево декомпозиции, наследуется от Entity, `docs/full-vision/05_Architecture/Entity_Platform.md`) с ресурсами на узлах. Офисные агенты и LLM-оценка достижимости («реальный потолок», аналитические справки) относятся к более позднему срезу (roadmap M7–M8) — их в первом срезе нет физически, поэтому часть вопросов §7 просто не имеет предмета до тех пор. Абстракция `LLMProvider` (`backend/app/llm/provider.py`) уже готова принять такой провайдер, когда придёт время, — этот ADR её не трогает.

## Решение (Decision)

Из 7 открытых вопросов §7 решены 4, под скоуп первого среза (детерминированное ядро, без LLM):

1. **Формулы увязки (вопрос 1).** Первый срез считает только арифметику: сумма ресурсов детей vs родитель, сумма требований узла vs доступно, разрыв = разница. Никакой LLM-оценки «реального потолка» — это отдельный поздний срез.
2. **Единицы измерения ресурсов (вопрос 2).** `resource.kind ∈ {people, money, other}`: `people` — загрузка юнита в %, `money` — сумма в валюте, `other` — текстовое требование без числа (участвует в увязке как флаг «есть требование», не квантифицируется).
4. **Порог определённости (вопрос 4).** Автодетекция системой: узел «в тумане», если нет KPI ИЛИ владельца ИЛИ данных; «определён» при наличии всех трёх. «Бэклог» — единственное ручное состояние, ставит человек.
7. **Триггер пере-увязки (вопрос 7).** Пересчёт на каждое изменение узла/ресурса — дерево первого среза небольшое, пересчёт дешёвый.

3 вопроса отложены — без формулировки решения, чтобы не подменять будущее проектирование догадкой:

- **Вопрос 3** (права ролей на чужие узлы) — до среза аутентификации/ролей доступа. В первом срезе ролевые ярлыки — чистая семантика, без модели прав.
- **Вопрос 5** (версионирование справок) — до среза офисных агентов (M7–M8): справок как сущностей ещё нет.
- **Вопрос 6** (каталог навыков) — до среза навыков-воркфлоу (листовой слой Ф1.2): каталога ещё нет.

Альтернатива «сразу спроектировать под LLM-справки» отклонена: она заставила бы первый срез тащить недостижимую в MVP инфраструктуру (маршрутизация к офисным агентам, версионирование справок) ради вопросов, которые сам код первого среза не задаёт.

## Последствия (Consequences)

- Модель данных Goal Entity получает конкретные поля: `resource.kind` (people/money/other) на ресурсах узла, состояние определённости (авто «туман»/«определён» + ручной флаг «бэклог» поверх lifecycle Entity_Platform §14.3).
- Пересчёт разрывов — синхронная операция на каждое изменение; при появлении LLM-справок (M7–M8) потребуется отдельный ADR на асинхронный/периодический пересчёт, т.к. дорогие вызовы нельзя гонять на каждое изменение.
- Права доступа к узлам, версионирование справок и каталог навыков остаются открытыми — реализация первого среза не должна их домысливать; `Management_Model.md` §7.2 фиксирует это явно.
- `Management_Model.md` §3.1, §3.2, §3.3, §4 и §7 обновлены этим проходом, чтобы отражать решения выше (см. связанный docs-коммит).
