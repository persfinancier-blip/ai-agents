# ADR-0001: Локальный LLM-fallback для офисных советников

**Статус:** Принято (требование fallback) + Предложено (выбор конкретных моделей — отдельный спайк)
**Дата:** 2026-07-07
**Авторы:** владелец продукта + AI-агент разработки

## Контекст (Context)

Внутренним офисным советникам (агент-советник и контуры фронт/мидл/бэк, `docs/full-vision/02_Product/Management_Model.md` §4) нужен LLM. Основной путь — облачный провайдер (Anthropic через `app/llm/anthropic_provider.py`). Но облако недоступно в ряде реальных ситуаций: нет активной подписки/ключа, нет интернета, проблемы с VPN. Для этих случаев нужен **fallback на локальные модели**.

В качестве пробы был протестирован FreeDeepseekAPI — OpenAI/Anthropic-совместимый прокси, который гоняет запросы через веб-сессию DeepSeek Chat в сохранённом Chrome-профиле. Он работал для ручных экспериментов, но держится на хрупкой связке «браузерная сессия + недокументированный веб-бэкенд»: сессия протухает, вёрстка/поток меняются, надёжность непригодна для продуктового fallback. Репозиторий сохранён вне воркспейса (`c:/Dev/FreeDeepseekAPI`, remote: github.com/ForgetMeAI/FreeDeepseekAPI) как справочный материал.

Абстракция `LLMProvider` (`backend/app/llm/provider.py` + `factory.py`, провайдеры `anthropic`/`stub`) уже спроектирована так, что локальный провайдер добавляется рядом с облачным без изменения остального кода.

## Решение (Decision)

1. **Fallback на локальный LLM — явное требование архитектуры.** Советники должны уметь работать в деградированном режиме без облака; выбор провайдера — через `LLM_PROVIDER` и `factory.py`, как сейчас.
2. **Прокси-через-веб-сессию (FreeDeepseekAPI) отвергается** как механизм fallback — хрупкость несовместима с ролью резерва. Код убран из воркспейса, но сохранён снаружи.
3. **Конкретные модели — отдельный спайк.** Оценить 2–3 опенсорс-модели с локальным инференсом по критериям: качество рассуждений на русском, лицензия, требования к железу, простота развёртывания (Ollama/llama.cpp/vLLM или аналог). До спайка кандидатов не фиксируем, чтобы не вписывать устаревшие данные.

## Последствия (Consequences)

- Плюс: советники переживают отсутствие подписки/интернета; архитектура провайдеров остаётся честной абстракцией, а не обёрткой над одним SDK.
- Плюс: правило «LLM SDK только внутри `app/llm/<provider>_provider.py`» (CONTRIBUTING) получает второе практическое обоснование.
- Минус: локальный инференс потребует железа и отдельной настройки окружения; качество локальных моделей ниже облачных — деградация должна быть видимой пользователю (пометка в UI, чьё мнение сгенерировано резервной моделью — уточнить при реализации).
- Обязательство: спайк по моделям до начала реализации M7–M8 (офисные агенты), результат — обновление таблицы кандидатов и новый статус этого ADR.
