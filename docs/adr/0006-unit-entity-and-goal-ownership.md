# ADR-0006: Юнит как сущность и назначение владельца цели

**Статус:** Принято
**Дата:** 2026-07-16
**Авторы:** владелец продукта + AI-агент разработки

## Контекст (Context)

`Management_Model.md` §2 (Ф2) с самого начала называет юнита («человек или ИИ-сотрудник», Единая рабочая сила, PRD §28) третьим шагом потока управления — исполнитель назначает юнита и прикрепляет навык-воркфлоу. Но юнит до сих пор не был смоделирован как сущность: в прототипе (`data.ts`) есть `OsUnit` с `kind: human | digital | hybrid | team | dept`, а в реализованном Goal Entity (ADR-0002) владелец узла — это `entity.owner`, свободное текстовое поле платформы Entity (см. `Entity_Platform.md`), которое ADR-0002 §7.1 вопрос 4 использует как один из трёх критериев «определённости» узла (KPI + владелец + данные).

Открытие юнит-milestone (этот ADR + backend-срез, промпт #41 отдельным проходом) требует зафиксировать модель ДО кода: какие бывают юниты, как цель на юнита ссылается и что происходит с существующими текстовыми `owner`. Без этого backend-срез будет угадывать схему.

Решения ниже приняты владельцем 2026-07-16, до начала этого прохода.

## Решение (Decision)

1. **Unit — подтип Entity**, тем же паттерном, что `Goal`/`Kpi`/`Decision`: строка в базовой таблице `entity` + таблица `unit` с `entity_id` как PK+FK (Management_Model §6: новых моделей данных вне Entity Platform не создаётся).

2. **`unit.kind ∈ {employee, agent, external, device}`** — четыре **атомарных** типа исполнителя:
   - `employee` — внутренний сотрудник (человек), RU-ярлык для будущего UI «сотрудник»;
   - `agent` — ИИ-агент, RU-ярлык «агент»;
   - `external` — внешний работник (подрядчик/фрилансер, вне организационного контура), RU-ярлык «вне контура»;
   - `device` — робот/устройство, RU-ярлык «устройство».

   Team и department — **не kind**, а будущая **надстройка группировки** атомарных юнитов (отдельный ADR + отдельный слайс, заносится в BACKLOG этим проходом). `hybrid` из прототипа `OsUnit` **не переносится** в канон — атомарный исполнитель либо человек, либо агент, либо внешний, либо устройство; смешанные случаи не вводятся.

   Это **операционализирует** PRD §28 (который перечисляет команды/отделы/мульти-агентные команды как формы Единой рабочей силы) тем же способом, каким Management_Model §5 операционализирует PRD §37 — сам PRD этим проходом не редактируется, расхождение фиксируется здесь и в Management_Model §8 (раздел «Расхождения с PRD»).

3. **Владение целью: замена, не сосуществование.** `Goal` получает новое поле `unit_id` (nullable FK → `unit.entity_id`). Goal API перестаёт отдавать/принимать свободный текст `owner` — единственный канал назначения владельца цели теперь юнит. Сосуществование текстового `owner` и `unit_id` на Goal не вводится: это исключило бы ровно тот текст-мэтчинг, отказ от которого — ключевое решение 2026-07-16 (см. пункт 5).

4. **Определённость узла (ADR-0002 §7.1 вопрос 4).** Критерий «владелец» в тройке (KPI + владелец + данные) меняется на «назначен юнит»: `compute_definiteness` проверяет наличие `goal.unit_id`, а не текста в `entity.owner`.

5. **`entity.owner` остаётся базовым полем платформы Entity** — другие подтипы (Decision, Kpi) по-прежнему могут его использовать; только Goal перестаёт с ним работать. Исторические значения в БД не трогаются (ни удаления, ни перезаписи вне миграции ниже).

6. **Разовая миграция данных (выполняется в слайсе 2, backend-срез промпта #41):** для каждого различного непустого `entity.owner` среди целей — создать `Unit(kind=employee, name=<текст>)` и проставить `unit_id` на затронутых целях. Без потери данных и без продолжения текст-хаков после миграции: решение 2026-07-16 «никакого текст-мэтчинга после этого перехода» остаётся в силе как окончательное — миграция одноразовая, а не постоянный механизм синхронизации.

Альтернатива «оставить `owner` текстом и добавить `unit_id` как необязательную ссылку рядом» отклонена: она сохраняла бы два источника истины для одного и того же факта (кто владеет целью) и оставляла бы дверь для текст-мэтчинга, чего решение 2026-07-16 явно избегает.

Альтернатива «добавить `team`/`dept` как kind сразу» отклонена: группировка юнитов — самостоятельный концептуальный вопрос (кто входит, как агрегируются ресурсы/загрузка группы) с собственными последствиями для увязки (ADR-0002 §3.2); смешивать его с вводом атомарных типов исполнителя означало бы спроектировать группировку вдогонку, без отдельного обсуждения.

## Последствия (Consequences)

- Backend-срез (промпт #41, отдельный проход после мержа этого ADR): таблица `unit` (Entity-подтип), поле `goal.unit_id`, обновление `compute_definiteness`, удаление `owner` из Goal API, миграция данных по пункту 6.
- `Management_Model.md` §2, §6, §7.1 обновлены этим проходом (см. связанный docs-коммит) — отражают четыре атомарных kind и замену критерия «владелец» → «назначен юнит».
- Фронтенд (попап карточки цели, строка «отв. …») не меняется в этом милестоне — дожидается API-среза (промпт #41) и отдельного UI-милестона.
- BACKLOG.md: строка «Юнит-сущность для назначения владельца цели» закрывается (канон готов); заводятся follow-up-пункты — ADR + слайс группировки (team/dept), UI-милестон юнитов (список, назначение из попапа), увязка юнита как ресурса.

## Открытые вопросы

Зафиксировано как нерешённое — backend-срез (промпт #41) не должен домысливать:

1. **Жизненный цикл юнита** (PRD §31) — состояния, активация/деактивация, увольнение/вывод из эксплуатации устройства и т.п. Не решено.
2. **Модель группировки team/dept** — ~~состав группы, агрегация ресурсов/загрузки, отдельный ADR. Не решено.~~ Решено ADR-0007: группа — Entity-подтип `unit_group`, `department` (строгое дерево, один дом на юнит) и `team` (m2m-оверлей); `goal.unit_id` теперь ссылается на любую Сущность рабочей силы (юнит или группу). Агрегация ресурсов/загрузки группы и вложенность команд остаются отдельными открытыми вопросами (ADR-0007 «Открытые вопросы» пп. 1, 2).
3. **Рабочее пространство юнита** (unit workspace) — есть ли у юнита собственное представление/панель, что в ней. Не решено.
4. **Права и доступ юнитов** — пересекается с ADR-0002 §7.2 вопрос 3 (права ролей на чужие узлы), но для юнита как отдельной сущности не рассматривалось. Не решено.
5. **Юнит как ресурс** — связь с `OsResource.load` (Management_Model §3.2, загрузка юнита в %): использует ли ресурсный слой `unit_id` напрямую или остаётся отдельным полем на ресурсе. Не решено.
