# ADR-0008: Срок и важность/срочность цели

**Статус:** Принято
**Дата:** 2026-07-22
**Авторы:** владелец продукта + AI-агент разработки

## Контекст (Context)

`GoalPopup` (промпт №35) с самого начала несёт две dashed-заглушки без данных: секция «СРОК» и секция «ВАЖНОСТЬ · СРОЧНОСТЬ» — задел под матрицу Эйзенхауэра. Владелец (2026-07-15) отнёс их к бэкенду: сначала поля и миграция, потом UI. Этот ADR фиксирует схему до кодового среза (тот же порядок, что в ADR-0006/0007), слайс разбит на два промпта — №50 (бэкенд, этот ADR) и №51 (попап).

## Решение (Decision)

1. **Три новых nullable-поля на `Goal`:** `deadline: date | null`, `importance: bool | null`, `urgency: bool | null`.
2. **`deadline` — дата без времени.** Ни временной зоны, ни времени суток; сравнения и сортировки по календарному дню. Диапазон не ограничен — прошлые даты допустимы (ретроактивные цели существуют).
3. **`importance`/`urgency` — независимые булевы флаги, не составное поле.** Каждый — да/нет; квадрант Эйзенхауэра (делать/планировать/делегировать/отложить) **не хранится**, а вычисляется в UI из пары флагов на лету. Один источник истины на пару флагов, а не денормализованный квадрант.
4. **`null` ≠ `false`.** Для всех трёх полей `null` значит «не задано» (владелец ещё не смотрел на цель под этим углом), а `false` — осознанный ответ «неважно» / «несрочно» / без срока. Различение специально сохраняется в API (PATCH: поле отсутствует — не трогать; поле явно `null` — снять значение, тот же diff-style, что и у остальных optional-полей `patch_goal`).
5. **`compute_definiteness` не меняется.** Функция остаётся про «назначен владелец + есть измеримый KPI»; deadline/importance/urgency в неё не входят — цель может быть «определена» без срока и наоборот.

**Отклонённые альтернативы:** хранить готовый квадрант строкой (`eisenhower: str | null`) — денормализация, риск рассинхрона при независимом патче одного флага; `deadline` как `datetime` — избыточная точность, продукт оперирует днями, не временем; трёхзначный enum вместо двух bool (`unset | yes | no`) — эквивалентно `bool | None`, но менее нативно для pydantic/SQLAlchemy.

## Последствия (Consequences)

- Backend: три nullable-колонки на таблице `goal` (миграция без backfill — существующие цели получают `null` по всем трём); `GoalCreate`/`GoalPatch`/`GoalRead` расширяются тем же diff-style, что и у `unit_id`/`parent_id`; `compute_definiteness` не тронут.
- Фронтенд (промпт №51, отдельный слайс): секции «СРОК» и «ВАЖНОСТЬ · СРОЧНОСТЬ» в `GoalPopup` оживают — дата-пикер для `deadline`, два тумблера для `importance`/`urgency`; квадрант выводится в UI из пары флагов, не запрашивается отдельным полем.
- Открытый вопрос: должен ли `deadline` в будущем влиять на «определённость» (просроченная цель без прогресса — повод для fog?) — не решается в этом ADR, оставлено на будущий проход.

## Открытые вопросы (не домысливать в кодовом срезе)

1. Влияние `deadline` на `compute_definiteness` / визуальный «туман». Отложено.
2. Уведомления/эскалация по приближающемуся или просроченному сроку. Отложено, вне скоупа этого среза.
